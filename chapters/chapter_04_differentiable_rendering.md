# 第 4 章：从 3D 高斯到图像——可微分渲染为什么能成立

**本章核心问题**：第 3 章已经解释了为什么 primitive 最后选高斯，也解释了 3D 椭球怎样局部投成 2D 椭圆。现在真正的问题是：

> 怎么把这些 3D 高斯真正变成一张图？而且，怎么让这个过程仍然可微，让图像误差能反向流回高斯的中心、形状、透明度和颜色？

如果上一章解决的是“为什么是高斯”，这一章解决的就是：

> 高斯到底怎样沿着一条可计算、可加速、可反传的管线，变成屏幕上的像素。

---

## 一、先把整条渲染链看成四步

不要一上来就掉进实现细节。先看全图。

每个高斯最终变成像素，大体会经历四步：

```text
Step 1. 世界空间里的 3D 高斯
        -> 送到相机空间
        -> 投成 2D 椭圆 footprint

Step 2. 按深度组织这些 footprint

Step 3. 只让每个像素看“真的会影响自己”的那一小批高斯

Step 4. 把这些高斯按前后关系做 alpha blending
```

对应到更紧凑的形式，就是：

```text
3D 参数
 -> 投影
 -> 筛选
 -> 混合
 -> 图像
```

这条链里最关键的是两件事：

- 它必须快，不然就回到 NeRF 那种慢渲染
- 它必须可微，不然就没法训练

所以你要一直带着这两个目标往下看。

---

## 二、输入到底是什么，输出到底是什么

### 2.1 输入不是“一个场景”，而是一堆带参数的高斯

每个高斯最少有这几类参数：

- `mu`：3D 中心
- `Sigma`：3D 协方差，表示椭球形状
- `alpha`：不透明度 / 密度强度
- `color`：颜色

所以一个高斯你可以先粗略记成：

```text
G_i = {mu_i, Sigma_i, alpha_i, color_i}
```

一个场景就是很多个这样的高斯。

### 2.2 输出是一张图像

最终输出当然是一张图像：

```text
C[x, y] = 每个像素的 RGB 颜色
```

问题是：

> 从很多 3D 高斯到一张 2D 图像，中间到底该怎样连接，才能既正确又高效？

---

## 三、第一步：把 3D 高斯送到屏幕上

这一部分是整个渲染链的数学起点。

### 3.1 先做世界 -> 相机 的变换

如果高斯中心在世界坐标系里是 `mu_world`，那么在相机坐标系里就是：

```text
mu_cam = R @ mu_world + t
```

对应的协方差则是：

```text
Sigma_cam = R @ Sigma_world @ R.T
```

这两条式子你现在应该已经很熟了。

它们在说：

- 中心跟着相机变换走
- 椭球形状也一起被正确旋转过去

所以到这一步，每个高斯都已经变成了：

> 相机眼里的一个 3D 椭球云。

### 3.2 高斯中心的投影是精确的

如果相机坐标中的中心是：

```text
mu_cam = [X, Y, Z]
```

那它的像素坐标中心就是：

```text
u = fx * X / Z + cx
v = fy * Y / Z + cy
```

也就是说，2D footprint 的中心很好算。

### 3.3 真正难的是“3D 椭球怎么变成 2D 椭圆”

透视投影本身是：

```text
(X, Y, Z) -> (X/Z, Y/Z)
```

问题就在分母这个 `Z` 上。

这意味着：

- 投影不是纯线性变换
- 你不能像旋转那样，直接拿一个固定矩阵把 3D 协方差“精确扔到 2D”

如果到这里你觉得“是不是麻烦了”，恰好说明你抓到了关键处。

### 3.4 关键救场：局部线性化

虽然透视投影全局不是线性的，但高斯本来就很局部。

所以在每个高斯中心附近，你可以只看一个小邻域。

在这个小邻域里，投影可以近似为：

```text
dp ≈ J dx
```

这里：

- `dx`：3D 空间中相对高斯中心的小扰动
- `dp`：屏幕空间中的小扰动
- `J`：投影函数在高斯中心处的 Jacobian

这一步的几何意义非常值得记住：

> 我们没有说透视投影突然变成了线性；我们只是在说，对一个足够小的高斯来说，它附近的投影扭曲可以用一块局部切平面来近似。

### 3.5 于是 2D 协方差自然出来了

一旦有了局部线性关系：

```text
dp ≈ J dx
```

协方差就按标准线性传播公式走：

```text
Sigma_2d ≈ J @ Sigma_cam @ J.T
```

这就是 3DGS 里最核心的数学桥梁。

如果你只背这一条式子，很容易忘。

但如果你记住下面这句话，就很稳：

> 屏幕上的 2D 椭圆，不是凭空画出来的，而是 3D 椭球在当前视点附近经过局部线性化之后传播出来的 footprint。

### 3.6 一个最小画面

```text
3D 里：

      一个很小的椭球云
           ______
       _.-'      '-._
    .-'              '-.
    '-._            _.-'
         '--.___.--'

局部投影后：

      一个 2D 椭圆 footprint
         .-''''-.
      .-'        '-.
      '-.        .-'
         '-.__.-'
```

所以渲染阶段真正操作的，已经不再是“3D 高斯本体”，而是：

- 屏幕中心 `mu_2d`
- 屏幕协方差 `Sigma_2d`

也就是一个个 2D 椭圆高斯。

## 四、第二步：为什么必须关心深度顺序

如果每个像素只受一个高斯影响，那事情很简单。

可现实中，一个像素常常同时落在多个高斯的 footprint 里。

这时你就要回答：

> 它们谁在前，谁在后？谁先贡献，谁后贡献？

### 4.1 为什么不能无脑相加

如果直接把所有高斯贡献加起来，会忽略遮挡关系。

但真实视觉里，近处的东西应该挡住一部分远处的东西。

所以你不能把多个高斯当成完全可交换的贡献源。

### 4.2 Alpha blending 的物理直觉

最直观的理解是：

- 远处先提供颜色
- 近处再吸收/遮住一部分光
- 层层往前累积

可以把它想成下面这条链：

```text
背景光
 -> 穿过远处高斯
 -> 再穿过中间高斯
 -> 再穿过近处高斯
 -> 到达相机
```

所以顺序不是装饰，而是物理上的必要条件。

### 4.3 实现上通常怎么做

最直接的做法，就是按某种深度指标给高斯排序，然后按顺序混合。

在工程里，一个很常见的近似是：

```text
用高斯中心的深度 z 作为排序依据
```

这不是绝对精确，但通常够用，因为高斯本身足够局部。

所以大致流程是：

```text
先按深度把高斯排好
再让每个像素沿这个顺序混合相关高斯
```

### 4.4 为什么“排序不可微”没有把整个方法毁掉

这是一个很好的问题。

严格来说，离散排序本身不是光滑可微函数。

但工程上，3DGS 仍然工作得很好，原因通常是：

- 高斯参数在相邻迭代里变化不至于剧烈跳跃
- 排序关系大多数时候相对稳定
- 主体梯度仍能通过“投影 -> 高斯评估 -> blending”这条主链流回去

换句话说：

> 3DGS 依赖的是一条“工程上足够可微”的链，而不是每个细节都在严格数学意义上处处光滑。

这点很重要，因为工程上的“能训起来”常常比纯粹的“形式上全光滑”更关键。

---

## 五、第三步：为什么不能让每个像素遍历所有高斯

到这里，一个最大的工程瓶颈已经出现了。

如果图像有 `H * W` 个像素，高斯有 `N` 个，那么最朴素的做法是：

```text
每个像素
    遍历全部 N 个高斯
    计算贡献
```

复杂度会接近：

```text
O(H * W * N)
```

这在大场景里几乎必炸。

### 5.1 但高斯有一个天然优势：它是局部的

高斯值离中心越远衰减越快。

这意味着：

- 一个高斯不可能对整张图都产生显著影响
- 它通常只影响屏幕上的一小片区域

所以工程师很自然会想到：

> 既然 footprint 是局部的，那就别让每个像素看所有高斯，只看真的覆盖自己的那一小批。

### 5.2 Tile-based culling 的直觉

一个常见策略是把屏幕切成小块，比如 `16 x 16` 的 tiles。

```text
整个屏幕：

+----+----+----+
| t0 | t1 | t2 |
+----+----+----+
| t3 | t4 | t5 |
+----+----+----+
| t6 | t7 | t8 |
+----+----+----+
```

然后：

1. 对每个高斯，先算一个大致的 2D 包围盒
2. 看它会覆盖哪些 tile
3. 建立 `tile -> relevant gaussians` 的映射

这样每个 tile 在渲染时，只需要处理和自己有关的高斯。

### 5.3 为什么包围盒很好算

因为你已经有了 `Sigma_2d`。

而 `Sigma_2d` 对应的是一个 2D 椭圆。

你完全可以用一个近似的 `k-sigma` 半径去包住它，比如：

```text
3σ 范围 ≈ 覆盖绝大部分高斯质量
```

于是每个高斯都能很自然地变成一个局部 bounding box。

### 5.4 这一步的本质

Tile culling 的本质不是“优化技巧小抄”，而是：

> 高斯既连续又局部，这让它不仅数学上好处理，工程上也特别适合空间局部性优化。

这也是它比很多全局式表示更适合实时渲染的重要原因。

## 六、第四步：像素到底怎样把多个高斯混起来

现在我们已经有：

- 一个像素会被哪些高斯影响
- 这些高斯的顺序
- 每个高斯在该像素处的 2D 椭圆高斯值

剩下的问题就是：

> 这个像素最终颜色怎么从这些局部贡献里长出来？

### 6.1 单个高斯在一个像素的贡献

对某个像素 `p`，先算它相对高斯中心的偏移：

```text
d = p - mu_2d
```

然后用二次型评估它在椭圆几何下离中心有多远：

```text
q = d.T @ inv(Sigma_2d) @ d
```

于是高斯值就是：

```text
g = exp(-0.5 * q)
```

再乘上高斯自己的 `alpha`，就得到这个高斯在该像素的有效权重。

### 6.2 混合的核心直觉：剩余透射率

一个很常见的写法是维护一个“剩余透射率” `T`：

- 开始时 `T = 1`，表示所有光都还能穿过
- 每混合一个高斯，就吸收一部分
- 后面的高斯只能拿到剩下那部分透射率

你可以把它想成：

```text
当前像素还能透过去多少“可见性预算”
```

每个高斯都会消耗一部分预算，并留下自己的颜色贡献。

### 6.3 为什么可以早停

如果某个像素在前面几个高斯之后，透射率已经很小了，比如接近 0，那么后面的高斯几乎不可能再贡献可见差异。

于是就可以提前停。

这又是一层很实用的工程优化：

- 不是所有相关高斯都要算到底
- 很多像素其实在前几层就已经“看不透了”

所以真实渲染里，平均每像素真正混合的高斯数通常远小于理论上界。

## 七、为什么整条链还能反向传播

到这一步，我们终于可以回答“可微分渲染为什么成立”了。

### 7.1 先看梯度的主干路径

图像损失 `L` 要往回流，大致会经过这条链：

```text
L
 -> 像素颜色 C
 -> blending 权重
 -> 2D 高斯值
 -> mu_2d, Sigma_2d
 -> mu_cam, Sigma_cam
 -> mu, Sigma, alpha, color
```

所以真正重要的是：

- 2D 高斯评估可微
- blending 可微
- 协方差传播可微
- 世界到相机的变换可微

这几件事一旦成立，参数就能被图像误差驱动更新。

### 7.2 哪些部分是“很自然地可微”的

下面这些几乎都很顺：

- 矩阵乘法
- 向量加法
- 指数函数
- 二次型
- 大部分 blending 链路

也就是说，3DGS 之所以能训，不是因为发明了某个神秘求导技巧，而是因为：

> 它的核心计算链大部分都建立在线性代数和光滑函数之上。

### 7.3 真正的麻烦点只有少数几个

主要是：

- 排序本身并不严格光滑
- 数值稳定性必须小心处理
- 协方差不能变成奇异矩阵
- 深度太小会让 Jacobian 爆炸

这时工程上通常做的不是放弃，而是：

- 对深度加下界裁剪
- 给 `Sigma_2d` 加小正则项
- 用更稳的方式处理二次型或逆矩阵
- 让大高斯在训练中被拆分，保持局部线性化有效

所以这条链的真实关键词不是“纯理论上无懈可击”，而是：

> 数学上主体可微，工程上再把不稳定点一个个钉住。

## 八、为什么它能比 NeRF 快这么多

这部分不是为了做性能吹嘘，而是为了看清“计算结构到底变了什么”。

### 8.1 NeRF 慢在哪

NeRF 的经典痛点是：

```text
对每条光线
    沿深度采很多点
    每个点都查网络
    再积起来
```

也就是大量采样 + 大量 MLP 查询。

慢的本质不是“实现没优化好”，而是它的计算结构天然就重。

### 8.2 3DGS 快在哪

3DGS 换了一条结构：

```text
先把 3D primitive 直接投成 2D 局部 footprint
再只在 footprint 覆盖区域里做混合
```

这相当于把大量“沿射线积分”的工作，改成了“在屏幕上处理局部椭圆贡献”。

这带来三个巨大差别：

1. 不再需要每像素沿深度采样几十上百次
2. 不再每个采样点都过神经网络
3. 高斯 footprint 局部性极强，非常适合 tile 筛选和 GPU 并行

所以快的根源不是单点技巧，而是：

> 整个表示和渲染方式，都被改写成了更适合实时图形管线的形式。

## 九、把整章压成一条完整主线

如果你只想记最短版本，就记这条：

```text
3D 高斯先在世界空间里表示为一个椭球云

    ↓

通过 R @ mu + t, R @ Sigma @ R.T 送到相机空间

    ↓

透视投影全局非线性，但对小高斯可局部线性化

    ↓

于是 3D 协方差用 J @ Sigma_cam @ J.T 传播成 2D 协方差

    ↓

每个高斯在屏幕上变成一个 2D 椭圆 footprint

    ↓

按深度组织，只在局部 tile 内处理相关高斯

    ↓

对每个像素做 alpha blending，得到最终图像

    ↓

因为主体链路可微，图像误差就能反向流回高斯参数
```

这就是“为什么 3DGS 既能渲染，又能训练”的核心解释。

## 十、本章你真正应该能自己重建的几个问题

读完这一章，你最值得会的不是背代码，而是自己把下面这些问题推回来：

1. 为什么 3D 椭球投到屏幕上会变成 2D 椭圆 footprint？
2. 为什么透视投影明明非线性，却还能写出 `J @ Sigma_cam @ J.T`？
3. 为什么排序是必要的，而不是可有可无的实现细节？
4. 为什么 tile-based culling 不是“小优化”，而是实时渲染成立的关键？
5. 为什么 alpha blending 不是简单求和？
6. 为什么整条链大部分都能反向传播？
7. 为什么 3DGS 的高效，本质上来自计算结构的改变，而不只是“CUDA 写得更好”？

如果这些问题你能遮住答案后自己说清楚，这一章就真的进脑子了。
