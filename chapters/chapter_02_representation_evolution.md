# 第2章：从体素到点云 - 表示方法的演变

**学习路径**：`starting point → invention`

---

## 回顾：我们站在什么基础上？

上一章我们提炼了核心矛盾：
- 需要**快速查询**（体素/点云的优势）
- 需要**连续渲染质量**（NeRF的优势）
- 但**体素太密集，点云无法渲染**

本章分析表示方法的演变逻辑，理解每一步的"被迫性"。

---

## 方案演进史：必然的探索路径

### 1. 体素时代：朴素的离散化

**历史背景**：早期3D重建（2000s-2010s）受计算机图形学体绘制（Volume Rendering）影响

**核心操作**：
```
对于每个ray，沿方向t采样：
  查询体素网格：V(x, y, z) → (occupancy, color)
  累加颜色：C += T * α * color
```

**为什么体素是"第一反应"？**
- 2D图像本身就是像素网格 → 3D用体素是自然推广
- 空间划分简单（均匀网格）
- 查询是O(1)数组索引 → 极快

**但体素的根本问题**：
- **内存墙**：三维空间离散化导致内存需求爆炸
  - 256³ = 16.7M体素（可行但粗糙）
  - 512³ = 134M体素（内存占用 ~4GB @ float32）
  - 1024³ = 1.07B体素（不可行）
- **边界模糊**：体素是"块状"的，表面呈锯齿状
- **渲染速度仍不足**：虽然查询快，但ray marching采样点多（100+次/ray）

**体素的遗产**：
- ✅ 证明了离散表示 + 预计算存储的可行性
- ✅ 确立了"渲染 = 投影 + 累加"的基本范式
- ❌ 但离散化本身不可扩展

---

### 2. 点云革命：稀疏性的觉醒

**历史转折**（2010s后期）：激光雷达普及 + SfM（Structure from Motion）成熟

**核心洞察**：
- 大部分3D空间是**空的** → 为什么存储空体素？
- 只存**表面点**即可，内存降至O(表面点数)

**点云表示**：
```
P = { (x_i, y_i, z_i, c_i) | i = 1..N }
其中 N << 体素总数，通常10⁵-10⁶点
```

**点云的渲染尝试**：
1. **点精灵（Point Sprites）**：每个点画一个面向相机的方块
   - 结果：点之间有空隙，不连续
2. **Splatting**：每个点画一个高斯模糊圆
   - 结果：有点覆盖，但仍是"贴纸"，没有3D厚度
3. **表面重建**：用Poisson重建 Mesh，再渲染
   - 结果：质量提升，但过程复杂且丢失细节

**点云的优势**：
- ✅ 内存效率：只存表面，N ~ 10⁵-10⁶ vs 体素 ~ 10⁸
- ✅ 天然稀疏：适合真实场景（大部分空间空）
- ✅ SfM可直接输出

**点云的根本缺陷**：
- ❌ **零维度**：点没有面积，无法直接"覆盖"像素
- ❌ **需要后处理**：Splatting/Mesh重建都是补救措施
- ❌ **视角依赖**：点精灵/高斯贴纸永远是面向相机的，不是真实3D

**点云的遗产**：
- ✅ 证明了**稀疏性**是可行且必要的
- ✅ 证明了**直接投影**（无需ray marching）可以快速
- ❌ 但"零维"本质限制了质量

---

## 关键洞察：什么被点云"几乎做对"但差了最后一步？

点云的Splatting尝试已经接近答案：
- 每个点画一个**高斯模糊圆**（2D）
- 多个高斯**叠加**形成连续图像

**问题在哪？**
- 这个"高斯"是**2D屏幕空间**的 → 不是真3D
- 它只是"贴纸"，没有**3D几何约束**
- 不同视角下，同一个3D点应该投影到不同大小的2D高斯 → 点云方法做不到

**最后的突破口**：
> 如果我们在**3D空间**就定义好每个点的"影响范围"（一个3D高斯椭球），让它自然投影到2D，会怎样？

这就是3D Gaussian Splatting的核心发明。
