## 一、问题：3D重建到底在干嘛？

### 1.1 一句话定义

从多张照片里重建出3D场景，然后能像玩游戏一样从任意角度渲染新照片。

**核心公式**（视图合成）：
```
已知：M张图片 + 每张的相机参数 {K_i, R_i, T_i}
目标：找到一个场景表示S
要求：用S + 新相机参数 {K, R, T} = 渲染出接近真实的新图片 C_pred ≈ C_gt
```

**魔鬼细节**：这个"Render"函数怎么写？不同的S导致完全不同的实现方式。

## 二、体素：第一反应，但很快撞墙

### 2.1 体素在干嘛？

**思路很简单**：把3D空间切成小立方体，每个存颜色和密度。

```
3D空间  →  V×V×V 体素网格
           ↓
每个体素存值
           ↓
Ray marching: 沿射线采样
           ↓
三线性插值读取
           ↓
累加颜色
```

**数学**（带变量注释）：
```
体素函数 V(x,y,z)  # 在体素网格中存储密度和颜色
  V[x,y,z].density ∈ ℝ⁺  # 该体素的密度
  V[x,y,z].color ∈ [0,1]³ # 该体素的颜色

射线参数化: r(t) = O + t·d  # O:光心(3,), d:方向(3,), t:距离标量
渲染积分（体渲染）:
  C = ∫₀^∞ T(t) · α(r(t)) · c(r(t)) dt
  其中:
    T(t) = exp(-∫₀ᵗ σ(s)ds)  # 透射率（剩余光线比例）
    α(r(t)) = 1 - exp(-σ(r(t))·Δt)  # 采样段的不透明度
    c(r(t))  # 采样点的颜色

离散化（ray marching）:
  C ≈ Σᵢ T(tᵢ) · αᵢ · cᵢ · Δt
  其中:
    tᵢ = t_near + i·Δt  # 第i个采样位置
    αᵢ = 1 - exp(-σᵢ·Δt)  # σᵢ = V[x_i,y_i,z_i].density
    cᵢ = V[x_i,y_i,z_i].color
    Δt = (t_far - t_near) / N_samples
```

**为什么当时选它？**
- 查询O(1)（直接索引）
- 结构规整，实现简单
- 预计算存储，实时查询 - 听起来很美

---

### 2.2 内存墙：立方增长的数学必然

**数字说话**（带计算）：

```
V=256  →  V³ = 256³ = 16,777,216 体素  →  64 MB  (假设4字节/体素)
V=512  →  512³ = 134,217,728 体素   →  512 MB
V=1024 →  1024³ = 1,073,741,824 体素  →  4 GB
V=2048 →  2048³ = 8,589,934,592 体素  →  32 GB  ❌

计算: mem = V³ × bytes_per_voxel
     假设 bytes_per_voxel = 4 (密度float) + 12 (颜色RGB) = 16字节
     则 V=256: 16.7M × 16B = 268 MB（更现实的估计）
```

**为什么是V³？**
- 3D空间，每个维度有V个离散单元
- 总单元数 = V × V × V = V³（立方增长）
- 这是数学必然，不是实现问题

**增长有多恐怖**：
```
V提高2倍  →  内存增加 2³ = 8倍
V提高4倍  →  内存增加 4³ = 64倍

举例:
  256 → 512 (×2) : 64MB → 512MB (×8)
  512 → 1024 (×2): 512MB → 4GB (×8)
```

消费级GPU最大24GB，V=2048直接超标。**体素分辨率有硬上限**，无法用于大场景。
```

---

#### 🔬 **第一性原理：为什么体素是"第一反应"？**

**公理**：
1. 3D空间可离散化
2. 渲染需要快速查询
3. 实时系统需要O(1)

**矛盾**：
- 快速查询 → 需要数组索引 → 规则网格
- 规则网格 → 内存V³
- 内存有限 → V不能大 → 质量差

**结论**：体素在"快速查询"约束下是必然选择，但内存墙无法突破。

---

### 2.3 质量天花板：锯齿无法避免

**问题**：离散网格 + 插值 → 边界必然锯齿。

```
理想表面:  ~~~~~~~~~~~~~~~~~~~~~~
            (连续曲线)

体素表面:  `````````````````````
            (离散台阶)
```

**本质**：连续函数用离散网格近似，必然有量化误差。V越大越平滑，但V受内存限制。

**体素的遗产与局限**：
- ✅ 证明了"预计算 + 快速查询"范式可行
- ✅ 确立了"渲染 = 投影 + 累加"框架
- ❌ 内存立方增长 → 不可扩展
- ❌ 边界锯齿 → 质量天花板低

**历史定位**：第一反应，死胡同。它的遗产是范式，教训是内存墙。

## 三、点云：发现99.9%空间是空的

### 3.1 核心洞察：稀疏性

**关键数据**：
```
3D空间占用率:
  空气/空区域: 99.9%
  物体表面:   0.1%
```

**推论**：为什么要存空体素？只存表面点！

**内存对比**：
- 体素: V³ (512³ = 134M)
- 点云: N (10⁵-10⁶)
- **减少100-1000倍** ✅

**这是范式转变**：从"完整空间表示"到"只存可见表面"。

---

#### 🔬 **第一性原理：稀疏性的必然性**

**公理**：
1. 真实场景物体只占小部分空间
2. 内存稀缺
3. 需要高质量渲染

**矛盾**：
- 体素存所有空间（包括空的）→ 浪费
- 但怎么知道哪里是空的？→ 需要先有表示

**解决路径**：
- SfM直接输出点云 → 已经是稀疏表面采样
- 决策：只存表面点
- 优点：内存O(N) << O(V³)
- 风险：点零维度，无法直接渲染

**验证**：从"内存有限+场景稀疏"能推导出只存表面吗？
- 内存有限 → 不能存所有体素
- 场景稀疏 → 大部分体素空
- 只存非空 → 需要先知道哪里非空 → SfM给了点云
- **结论**：点云是体素内存墙的必然逃生方案

---

### 3.2 点云的"三次尝试"：怎么给点加体积？

点云知道"哪里有表面"，但不知道"表面有多大"。三次尝试都在解决这个问题。

---

#### 尝试1: 点精灵 (Point Sprites)

```
方法: 每个点画一个面向相机的方块

问题:
  • 点间有空隙 → 不连续
  • 方块角度固定 → 不是真3D
  • 放大见方块边界

结果: ❌ 质量差
```

**诊断**：方块有面积，但**面积固定不随视角变** → 不是真3D体积。

---

#### 尝试2: Splatting (2D高斯)

```
方法: 每个点画一个2D高斯圆
  公式: G_2d(x) = exp(-|x - μ_2d|² / (2σ²))
  其中:
    x: 像素坐标 (2,)
    μ_2d: 点投影到屏幕的中心 (2,)
    σ: 高斯半径（像素，固定或根据深度调）

问题:
  • 2D屏幕空间 → 不是真3D，无几何约束
  • 只是贴纸，不同视角无法保持一致性
  • 同一点在不同距离应投影到不同大小 → 2Dσ无法表达

关键困境: σ怎么定？
  方案1: 固定σ（如3像素）
    → 近处点：高斯显得太小（透视错误）
    → 远处点：高斯显得太大（透视错误）
    ✗ 结果：深度相关伪影

  方案2: σ ∝ 1 / depth（根据深度调整）
    → 需要知道该点在3D空间的"真实体积"
    → 点云没有体积信息！ ✗
    → 只能瞎猜，无法学习

诊断: 2D高斯缺少**3D几何约束**
  - 无法表达"这个点在3D空间有多大"
  - 不同视角的投影不一致（因为缺少3D→2D的投影公式）
  - 需要真正的3D体积表示
```

---

#### 尝试3: Poisson重建 → Mesh

```
方法: 点云 → Poisson → Mesh → 三角形渲染

结果:
  ✅ 质量提升
  ❌ 过程复杂 (Poisson慢)
  ❌ 丢失细节 (简化)
  ❌ 无法端到端优化
```

**诊断**：Mesh质量好，但**失去稀疏性和可微性**，是退步。

---

### 3.3 点云的"最后一公里"问题

**缺口**：

```
点云: 位置 + 颜色
   需要: 每个像素有3D元素负责 → 需要面积
   需要: 连续投影 → 需要连续分布
   需要: 直接投影 → 需要解析公式
   没有: 影响范围（体积）
```

**突破点思考**（eureka moment）：

```
如果给每个点定义一个"影响范围"（椭球），
让它自然投影到2D，会怎样?

3D点 + 3D椭球  →  投影  →  2D高斯
        ↑
    这就是3DGS的核心
```

**为什么是椭球？候选形状对比**：

```
形状    | 投影复杂度 | 连续? | 各向异性?
--------|------------|-------|----------
立方体  | 复杂       | ❌    | ✅
球体    | 简单       | ✅    | ❌ (只能各向同性)
椭球    | 中等       | ✅    | ✅
高斯    | 简单       | ✅    | ✅ (协方差)
```

**高斯是唯一解？** 下一章详细推导。

---

**点云的遗产**：
- ✅ 证明稀疏性可行（内存减100倍）
- ✅ 证明直接投影可以快（无需ray marching）
- ❌ 遗留：零维度 → 需要给点加体积

## 四、NeRF：连续表示的代价

### 4.1 NeRF在做什么？

**核心想法**：用神经网络表示连续场景。

```
F_θ(x, y, z, direction) → (color, density)
渲染: C(r) = ∫ T(t)·σ(t)·c(t) dt
训练: L2(渲染, GT) → 反向传播更新θ
```

**优点**：
- 连续表示，质量好
- 视角依赖效果（镜面反射）建模成功
- 端到端可微

**但速度...**

---

### 4.2 NeRF的"三宗罪"

#### 罪1: MLP查询慢（无法避免）

**复杂度分析**（带变量注释）：

```
每ray采样数: N_s ≈ 100-200  # 沿一条射线的采样点数量
每样本MLP FLOPs: ≈ 10,000   # 5层×64神经元，每层约2000 FLOPs
每ray总FLOPs: N_s × 10⁴ ≈ 10⁶ FLOPs  # 单射线计算量

图像ray数: 1920×1080 ≈ 2.07M  # 像素数 = 射线数
总FLOPs/帧: 2.07M × 100 × 10⁴ = 2×10¹² FLOPs = 2 TeraFLOPs!

理论下限 (RTX 4090, 80 TFLOPS):
  2 / 80 = 0.025秒 = 25ms

实际NeRF: 30秒/帧
差距: 约1200倍（原因：内存带宽瓶颈、kernel启动开销、非峰值算力）

为什么MLP不能缓存？
- 每个ray采样点位置x不同 → MLP必须重算
- 视角d变化 → 输出变化 → 无法重用
- 3D信息编码在MLP权重θ中，不是显式结构
- **结构性缺陷**：表示（MLP）与查询（每点重算）耦合，无法分离
```

**NeRF体渲染公式**（完整版）：
```python
# 输入: 射线 r(t) = O + t·d
# 输出: 像素颜色 C

T = 1.0  # 透射率
C = 0.0  # 累计颜色

for i in range(N_s):
    t = t_near + i * Δt
    x = O + t * d  # 3D采样点
    
    # MLP查询（瓶颈！）
    σ, c = MLP_θ(x, d)  # σ:密度(标量>0), c:颜色(3,)
    
    α = 1 - exp(-σ * Δt)  # 不透明度
    
    C += T * α * c  # 前向混合
    T *= (1 - α)    # 透射衰减
    
    if T < 1e-4: break

return C
```

---

#### 罪2: ray marching本质离散

**采样-质量权衡**（无解）：

```
N_s ↑ → 质量↑ 速度↓
N_s=64:  快但噪点多
N_s=128: 平衡
N_s=256: 质量好但慢2倍
```

**为什么必须采样？**
- 连续函数f(x)的积分无解析解（f是MLP）
- 必须数值积分 → 离散采样
- **数学必然，无法避免**

---

#### 罪3: 训练时间长

```
每ray 1ms → 每epoch 200秒
30k步 ≈ 1500 epochs → 300,000秒 ≈ 83小时
(实际有优化, 但仍需数小时-天)
```

**原因**：每帧都要跑完整管线，每样本都要MLP查询。

---

#### 🔬 **第一性原理：NeRF瓶颈的根源**

**公理**：
1. 场景是连续函数（MLP输入x,d输出color,density）
2. 渲染需要沿射线积分
3. MLP是通用函数逼近器

**矛盾**：
- MLP每帧必须重算 → 无法缓存3D结构
- 采样数N_s与质量正相关，速度负相关
- 训练需要大量射线

**解决路径**：
```
问题: MLP慢 + ray marching慢
思路: 能否避免ray marching？能否避免每样本MLP？

方向1: 预计算3D结构 → 缓存 (Instant-NGP: hash grid)
方向2: 改变表示 → 可直接投影的primitive (3DGS: 高斯)
```

**验证**：从"连续表示+实时渲染"推导：
- 连续表示 → 需要积分 → 需要采样
- MLP → 每样本计算 → 无法缓存
- 能否预计算？ → Instant-NGP尝试，仍有查询开销
- **结论**：NeRF的瓶颈是"连续函数+实时查询"的**结构性矛盾**

---

**NeRF的遗产**：
- ✅ 证明神经连续表示可行
- ✅ 视角依赖效果建模成功
- ✅ 端到端可微范式
- ❌ 局限：MLP瓶颈、离散采样、无缓存 → 必须突破

### 5.2 为什么是"高斯"？形状选择决策树

```

        |
        +---> 立方体?  → 投影复杂，边界不连续 → ✗
        |

**高斯的三大优势**：

2. ✅ **形状完整**：协方差矩阵描述任意椭球（尺度+旋转）
3. ✅ **数学优良**：可微、旋转不变、中心极限定理

### 2. 投影公式推导（带注释）

**问题**：假设3D高斯Σ是各向同性的（σ²I），投影到2D后Σ₂D是什么形式？

**推导**（简化版，假设相机看向+Z轴）：

```
# 3D高斯（球对称）
Σ = σ² · I₃  # I₃是3×3单位矩阵，σ是尺度参数（标量）

# 投影操作：透视投影近似（忽略z缩放）
# 实际投影公式：Σ_2d = J · Σ · Jᵀ
# 简化假设：J ≈ [1/z, 0, 0; 0, 1/z, 0]（忽略∂x/∂Z项）
# 则:
Σ_2d = (1/z²) · Σ[:2, :2]  # 取Σ的左上角2×2子矩阵
      = (1/z²) · σ² · I₂
      = (σ/z)² · I₂

结论:
  各向同性3D高斯（σ²I） → 各向同性2D高斯（(σ/z)²I）
  2D尺度 σ_2d = σ / z

物理意义:
  3D尺度σ，除以深度z，得到2D屏幕尺度
  近处物体（z小）→ σ_2d大 → 投影大
  远处物体（z大）→ σ_2d小 → 投影小
  ✅ 符合透视几何！
```

**一般情况（Σ是椭球）**：
```
Σ = diag(σ_x², σ_y², σ_z²)  # 各向异性，主轴对齐坐标轴
Σ_2d = diag(σ_x²/z², σ_y²/z²)  # 丢弃Z方向
      = diag((σ_x/z)², (σ_y/z)²)

结论:
  3D椭球 → 2D椭圆
  2D各轴尺度 = 3D各轴尺度 / z
  保留了各向异性（X和Y方向可以不同）
```

**关键洞察**：
- 投影是**线性变换**（去掉Z维 + 缩放）
- 正态分布在线性变换下仍是正态分布（投影封闭性）
- 这就是为什么高斯是唯一解：其他分布（如均匀分布）投影后不是原分布

**需要独立重推吗？** 试试从"3D椭球 + 透视投影"出发，自己推导Σ_2d的公式。
```

### 4. 为什么其他分布不行？

**问题**：尝试用均匀分布代替高斯，投影公式会怎样？

```
3D均匀分布: 椭球内均匀，外部0
投影: 需要计算椭球在2D的截面面积
      → 截面形状复杂（椭圆，边界不规则）
      → 无解析公式 ❌

高斯: 投影仍是高斯 ✅ (唯一性)

结论: 高斯是唯一满足"3D→2D投影封闭"的常见分布
```

**数学验证**：
- 正态分布的线性变换仍是正态分布
- 投影是线性变换（去掉维度）→ 仍是正态分布
- 其他分布（均匀、指数）不满足
