## 二、动态场景：4D Gaussian Splatting

### 2.1 Problem：静态假设失效

**问题**: 3DGS假设场景静态，但真实世界有运动

```
+----------------+----------------+----------------------+
| 场景           | 运动元素       | 静态3DGS问题         |
+----------------+----------------+----------------------+
| 城市街景       | 行人、车辆、树叶| 高斯无法表示运动      |
| 人脸表情       | 肌肉运动       | 几何固定             |
| 室内交互       | 开门、物体移动 | 无法编辑             |
+----------------+----------------+----------------------+
```

**量化影响**:
- 如果场景有30%动态区域，静态3DGS重建误差增加5-10 dB

---

### 2.2 Invention：时变高斯参数

**核心思想**: 每个高斯参数是时间t的函数

```
μ(t) = μ₀ + Δμ(t)
Σ(t) = Σ₀ + ΔΣ(t)
α(t) = sigmoid(α₀ + Δα(t))
c(t) = c₀ + Δc(t)  # 可选
```

**实现方案对比**:

```
+--------+----------+----------+----------+----------+----------+
| 方案   | 时变建模 | 内存开销 | 速度     | 质量     | 适用场景 |
+--------+----------+----------+----------+----------+----------+
| 线性插值| 关键帧间线性| 低(×2)  | 快       | 中       | 简单运动 |
| 多项式 | 三次/五次多项式| 中(×3-4)| 中       | 良       | 连续运动 |
| 小MLP  | 神经网络预测| 高(×10+)| 慢       | 优       | 复杂运动 |
+--------+----------+----------+----------+----------+----------+
```

---

### 2.3 代表工作：4D Gaussian Splatting

**方法**:
- 每个高斯存储: μ₀, Σ₀, α₀, c₀ + 时间基函数系数
- 时间基: 多项式（如三次）
- 任意时刻t: 计算基函数值 → 得到该时刻参数

**性能**:
- 内存: 从52MB → 150MB（3倍）
- 速度: 仍可实时（30-60 FPS）
- 质量: PSNR比静态3DGS低1-2 dB（因参数共享）

---

### 2.4 Decision Tree：何时用4D GS？

```
场景有动态?
    |
    +---> 否 → 用静态3DGS
    |
    +---> 是 → 运动复杂度?
              |
              +---> 简单 (平移/旋转) → 关键帧+线性插值
              |                    内存×2, 速度✓
              |
              +---> 中等 (连续变形) → 多项式建模
              |                    内存×3-4, 速度△
              |
              +---> 复杂 (非刚性) → 小MLP预测
                                   内存×10, 速度✗
```

## 三、压缩与流式传输

### 3.1 Problem：数据量太大

**原始大小**:
- 1M高斯 × 13 float × 4字节 = **52 MB**
- 3M高斯 = **156 MB**

**场景**:
- 移动端存储有限（<100MB场景）
- 网络传输（5G/4G带宽有限）
- 流式加载（首帧快速）

**需求**: 压缩至 **<10 MB** 同时保持质量（PSNR下降<1 dB）

---

### 3.2 Invention：量化 + 熵编码 + 渐进式

#### 方案1：参数量化

**量化策略表**:

```
+--------+----------+----------+----------+----------+
| 参数   | float32  | 量化后   | 压缩比   | PSNR损失 |
+--------+----------+----------+----------+----------+
| μ(位置) | 4字节    | int16(2字节)| 2×     | <0.1 dB  |
| Σ(尺度) | 4字节    | uint8(1字节, log空间)| 4× | 0.2 dB |
| α       | 4字节    | uint8(1字节)| 4×     | <0.1 dB  |
| c       | 4字节    | uint8(1字节)| 4×     | 0.3 dB   |
+--------+----------+----------+----------+----------+
| 总计   | 52 MB    | ~13 MB   | 4×       | <0.5 dB  |
+--------+----------+----------+----------+----------+
```

**量化实现**:
```python
# μ: 归一化到[-10,10] → int16
mu_norm = (mu - scene_min) / (scene_max - scene_min)  # [0,1]
mu_quant = (mu_norm * 65535 - 32768).short()  # int16

# Σ: log空间 → uint8
log_scale = torch.log(scale)
log_min, log_max = log_scale.min(), log_scale.max()
scale_quant = ((log_scale - log_min) / (log_max - log_min) * 255).byte()

# α, c: 线性映射 [0,1] → [0,255]
alpha_quant = (alpha * 255).byte()
color_quant = (color * 255).byte()
```

---

#### 方案2：差分编码 + 熵编码

**空间相关性**: 相邻高斯位置相近 → 差分Δμ小 → 可压缩

```python
# 排序后（按空间位置）
dmu = mu[1:] - mu[:-1]  # 差分
# 用arithmetic coding或zlib压缩dmu
```

**额外压缩**: 20-30%

---

#### 方案3：渐进式流式

**问题**: 首帧渲染需要所有高斯，但网络传输慢

**方案**:
1. 只传输视锥内高斯（culling）
2. 分块传输：先传中心，再传边缘
3. LOD：远距离用合并高斯

**效果**:
- 首帧：只传10%高斯，100ms内出图
- 后续：渐进补充

---

### 3.3 代表工作：Compressed 3DGS

**方法**:
- 混合量化（μ: int16, Σ/α/c: uint8）
- 空间差分 + zlib压缩
- 最终大小：5-8 MB（1M高斯）

**质量损失**: PSNR下降 < 0.5 dB（几乎不可见）

## 四、训练加速

### 4.1 Problem：训练仍太慢

**原始3DGS**: 30k步 ≈ 30分钟（RTX 4090）
**大规模场景**: 数小时

**需求**: 加速5-10倍，保持质量

---

### 4.2 Invention：稀疏训练 + 多分辨率

#### 方案1：稀疏视角训练

**观察**: 3DGS泛化能力强，不需要所有视角都看到

**实验**:
- 用10%视角训练 → PSNR下降 < 1 dB
- 用20%视角训练 → PSNR下降 < 0.5 dB

**策略**:
```
每epoch随机采样20%图像
等价于: 每5个epoch才完整遍历一次数据集
加速: 5×
```

---

#### 方案2：多分辨率渐进

**调度**:
```
0-10k步: 256×256（快速收敛）
10k-20k步: 512×384（中等）
20k-30k步: 1024×768（原分辨率）
```

**加速**: 约3×（结合稀疏训练可达15×）

---

### 4.3 代表工作：Fast 3DGS

**方法**:
- 稀疏训练（20%数据）
- 多分辨率（256→512→1024）
- 梯度累积（虚拟batch增大）

**结果**:
- 训练时间: 30分钟 → 6分钟
- PSNR损失: < 1 dB

---

## 五、几何质量提升

### 5.1 Problem：高斯可能"融化"

**现象**: 训练中高斯变得极扁（Σ接近奇异）

```
正常高斯:      ooooo
               o   o
               o   o
                oooo
               椭球形
               
"融化"高斯:    -
               -
               -
               -
              极扁，数值不稳定
```

**原因**:
- 梯度更新使Σ某些特征值→0
- 没有约束，数学上允许但物理上无意义

---

### 5.2 Invention：约束 + 正则化

#### 方案1：各向同性约束

**强制Σ = σ²·I**（球对称）

```python
def isotropic_loss(Sigma):
    diag = Sigma.diagonal(dim1=1, dim2=2)  # (N,3)
    mean_diag = diag.mean(dim=1, keepdim=True)
    target = torch.diag_embed(mean_diag.repeat(1,3))
    return F.mse_loss(Sigma, target)
```

**效果**:
- ✅ 防止Σ奇异
- ❌ 失去旋转自由度，PSNR -1~2 dB

---

#### 方案2：尺度上限裁剪

**硬约束**:
```python
max_scale = 1.0
Sigma.data.clamp_(max=max_scale**2)
```

**效果**:
- ✅ 简单有效
- ⚠️ 可能影响优化（梯度截断）

---

### 5.3 代表工作：Surf-GS

**方法**:
- 额外网络预测SDF（有符号距离场）
- 损失: L_surface = Σ distance(高斯中心, SDF表面)
- 约束高斯贴合几何表面

**效果**:
- ✅ 几何更干净
- ⚠️ 增加网络开销（训练稍慢）

## 六、与NeRF结合

### 6.1 Problem：3DGS的view-dependent效果弱

**3DGS局限**:
- 颜色c是常数（或随训练微调）
- 真实场景有视角依赖：镜面反射、透明物体、次表面散射

**NeRF优势**:
- 用视角相关MLP建模 → 捕捉view-dependent效果

**结合目标**: 3DGS的速度 + NeRF的视角效果

---

### 6.2 Invention：小MLP + 高斯

**思路**:
- 保持高斯几何（快速渲染）
- 颜色c由小MLP预测: c = MLP(μ, view_dir)

**公式**:
```
C(p, d) = MLP_φ(p, d)  # p=高斯中心, d=视角方向
```

**训练**:
- 可微分渲染仍通过高斯投影
- 但最终颜色来自MLP → 梯度更新MLP + 高斯参数

**MLP设计**:
- 输入: μ(3) + d(3) = 6维
- 隐藏层: 1-2层，每层32神经元
- 输出: RGB(3)
- 参数量: < 1K（几乎无开销）

---

### 6.3 代表工作：GaussianShader

**方法**:
- 每个高斯附加一个小MLP（共享权重）
- 训练时MLP和3DGS联合优化
- 推理: MLP前向 + 高斯渲染

**效果**:
- ✅ 获得view-dependent效果（镜面反射）
- ✅ 速度仍快（MLP推理<1ms）
- ⚠️ 训练稍复杂（两个模块）

**性能**:
- 渲染速度: 从80 FPS → 70 FPS（MLP开销）
- PSNR提升: +1~2 dB（在镜面区域）

## 九、本章总结

3DGS的核心思想（稀疏高斯 + 可微投影）已被证明具有强大扩展性。后续工作都在**保持核心简洁**的前提下，解决特定痛点。

**设计哲学**:
- ✅ 问题驱动：每个扩展针对具体痛点
- ✅ 最小改动：核心不变，加最小组件
- ✅ 保持可微：新增模块仍可微
- ✅ 性能意识：不牺牲实时性

**未来方向**:
- 更大场景（城市级 >100M高斯）
- 动态+编辑（实时可编辑）
- 端侧部署（手机/AR眼镜）
- 多模态融合（文本/音频控制）

---

**关键记忆点**:
- ✅ 动态: 时间参数化（多项式/MLP）
- ✅ 压缩: 量化（μ:int16, Σ/α/c:uint8）
- ✅ 加速: 稀疏训练 + 多分辨率
- ✅ 几何: 各向同性约束 + 表面对齐
- ✅ 结合: 小MLP预测颜色
- 🎯 **设计哲学**: 最小改动，保持可微，性能优先
