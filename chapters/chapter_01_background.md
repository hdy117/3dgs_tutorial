# 第 1 章：为什么我们需要 3DGS？——从第一性原理推导视图合成

**本章问题**：如果你只有几张照片，怎么"看到"没拍过的角度？

---

## 一、从一个简单的问题开始

想象这个场景：你用手机拍了房间四周的照片（大概 10-20 张），现在你想看**某个完全没拍过的角度**——比如蹲下来仰视天花板。你的直觉告诉你，这应该可以从已有照片"推断"出来，对吧？

但计算机不这么想。对计算机来说，每张照片只是像素矩阵——它不知道：
- 哪块像素是桌子、哪块是墙
- 这些物体在 3D 空间里是什么形状
- **新视角**会看到什么

这就是**Novel View Synthesis（视图合成）**问题的核心：**用有限照片重建"可渲染的 3D 场景"**。

### 问题形式化

```
输入：M 张照片 I₁, I₂, ..., Iₘ + 每张的相机参数 C₁, C₂, ..., Cₘ
输出：任意新视角 C_new 对应的图像 I_new
```

**关键约束**（这是问题的本质）：
1. **稀疏性**：我们只有几十到几百张照片
2. **稠密性要求**：要渲染的是连续、无锯齿的图像
3. **实时性需求**（可选但重要）：游戏/AR 需要 60fps

### 思考一下

如果让你设计一个方案，你会怎么表示"3D 场景"？

---

## 二、最朴素的想法：把世界切成方块

### 体素（Voxel）——体积像素

**直觉**：2D 图像由像素组成，那 3D 空间是不是可以由**体素**（3D pixel）组成？

```
每个体素存储：
- 颜色 c ∈ RGB
- 密度 σ > 0（表示这个位置有没有东西）
```

**渲染过程**：从相机发射射线，穿过体素网格，用**Volume Rendering Equation**积分：

```math
C = ∫ T(t) · σ(t) · c(t) dt, 其中 T(t) = exp(-∫₀ᵗ σ(s)ds)
```

这个公式的意思是：**沿射线的每个点贡献颜色，但被前面的东西遮挡（T是透射率）**。

### 体素的致命缺陷：内存爆炸

**公理**：如果空间分辨率翻倍，体素数量变成 8 倍（³维立方增长）。

算一笔账：
- **128³ 网格** = 209 万个体素 → 可接受
- **512³ 网格** = **1.34 亿个体素** → 每个存 RGB+σ = 512MB（还只是存数据）
- **1024³ 网格** = **10.7 亿个体素** → **需要 4GB 只存密度和颜色！**

**矛盾出现了**：
> 要高质量渲染 → 需要高分辨率体素 → 内存爆炸

而且大部分空间是空的（桌子后面、墙里面），我们付了钱却没用到。

### 自然演进：既然大部分是空的，为什么不全存？

→ **点云（Point Cloud）** 登场

---

## 三、点云：只存表面，但有个大坑

**想法很简单**：如果 90% 的空间都是空气，为什么不**只存储有东西的地方**？

```
每个点 pᵢ = (x, y, z, r, g, b)
```

**优势**：内存从 V³ 降到 N（N 是点的数量），而且 N << V³。

### 新问题：点是零维的

**想象一下渲染过程**：相机发射射线，怎么和点云交互？

- **理想情况**：射线刚好穿过某个点 → 获取颜色
- **实际情况**：99.9% 的射线从点的间隙中穿过 → **什么也看不到**

这就是**采样问题（Sampling Problem）**。点在 3D 空间是零维的，投影到 2D 也是零维的——它们没有"面积"来贡献像素颜色。

### 尝试修复：给点加体积？

**朴素方案 A：点精灵（Point Sprites）**
- 每个点渲染成一个小圆圈
- **问题**：近大远小关系不对，透视错误明显

**朴素方案 B：2D Gaussian Splatting**
- 直接在屏幕空间画高斯椭圆
- **问题**：没有 3D 几何约束，换个角度就穿帮

**核心矛盾**：
> 我们需要"有体积的东西"来贡献颜色，但点云只有位置。给点加体积的尝试都失败了——要么几何错误，要么视角不一致。

---

## 四、NeRF：另辟蹊径，用神经网络表示场景

### 关键跳跃：不用离散结构了！

**到这一步你可能在想**："体素有内存问题，点云有采样问题...怎么办？"

**NeRF（Neural Radiance Fields, 2020）的想法更激进**：
> 为什么不把整个场景存成一个**连续函数**呢？

```math
场景 = MLP 神经网络 θ(x, d) → (σ, c)
输入：3D位置 x ∈ ℝ³ + 视角方向 d ∈ ℝ²/³
输出：密度 σ > 0（这个位置有多"实"）+ 颜色 c ∈ [0,1]³
```

**渲染方式**：和体素一样，用 Volume Rendering Equation。但不用查表了——直接对任意 x 查询 MLP！

### 为什么这个方法质量好？

**公理**：MLP 是**通用函数逼近器（Universal Function Approximator）**——理论上可以表示任何连续函数。

这意味着：
- ✅ **无锯齿**：场景是连续的，不是离散的像素/体素
- ✅ **视角相关颜色**：输入 d 可以建模镜面反射、各向异性材质
- ✅ **无限分辨率**：想渲染多高分辨率都行（只要计算跟得上）

### NeRF 的"原罪"：慢到无法使用

现在问题来了。假设你要渲染一张 1920×1080 的图片...

#### 第一重灾难：每像素要采样上百次

**为什么？**
- Volume Rendering Equation 是积分，没有解析解
- 必须用数值方法（Ray Marching）离散近似
- **每个像素发射一条射线，每条射线采样 100-200 个点**

```
计算量估算：
- 像素数：1920 × 1080 ≈ 2M
- 每像素采样：100 次（保守估计）
- 总采样点：2亿个
- 每个采样点：查询 MLP（假设 4 层，每层 256 神经元）≈ 1万次 FLOPs
- **总算力需求**：2×10⁸ × 10⁴ = 2×10¹² FLOPs = **2 TeraFLOPs/帧**

对比 RTX 4090（80 TFLOPS）：
- 理论极限：2 / 80 = 25ms/帧
- 实际 NeRF：**30秒/帧**
- 差距：**1200倍**
```

**为什么差距这么大？** MLP 查询的内存访问不是连续的，GPU 利用率极低——但这还不是最致命的。

#### 第二重灾难：训练时间以天计

```
每张图片：2亿次MLP查询（上面算过了）
每个 epoch：假设10张图 × 200秒 = 2000秒
30k训练步 ≈ 1500 epochs → **83小时**（实际优化后也要数小时-天）
```

**原因**：每步训练都要做完整的 Ray Marching + MLP 前向。

### NeRF 的结构性瓶颈（第一性分析）

让我们回到公理层面：

| 公理 | 含义 |
|------|------|
| A1 | 场景 = MLP 表示的连续函数 |
| A2 | 渲染需要沿射线积分（Volume Rendering）|
| A3 | **MLP 是黑盒**——给定 x，必须前向传播才能知道 σ(x) |

**从这些公理能推出什么？**

> **推论**：每帧的所有采样点都要重新计算 MLP → **结构性慢，无法避免**

这不是工程问题。NeRF 的表示方式（隐式函数）决定了查询必须是 O(采样点数)。

## 五、3DGS 的关键跳跃：换表示

### "啊哈！"时刻

到这一步你看到两个方向：
1. **优化 NeRF**（Instant-NGP 等）：用 Hash Grid 缓存 MLP，加速查询
2. **换个思路**：如果不用 MLP 呢？

**3D Gaussian Splatting (2023) 选择了后者**。核心想法：

> "既然点云的问题是'点是零维的'，那为什么不直接给每个点一个**3D 椭球体积**（高斯分布），然后利用高斯的数学性质**直接投影到 2D**？"

### 为什么是高斯？——数学上的必然性

这里需要一点直觉。假设你在 3D 空间有一个高斯分布：

```math
G₃D(x) = exp(-½ (x - μ)ᵀ Σ⁻¹ (x - μ))
其中：μ是中心，Σ是协方差矩阵（决定椭球形状）
```

**关键性质**：3D 高斯经过透视投影后，在 2D 平面上**仍然是高斯**！

```math
2D 投影的参数有闭式解：
- μ_2d = K · (R·μ + T)      # 中心点直接投影（K是内参，R,T是外参）
- Σ_2d = J · (R·Σ·Rᵀ) · Jᵀ   # 协方差通过雅可比矩阵变换

其中 J = ∂(x₂d, y₂d)/∂(x₃d, y₃d, z₃d) 是投影的雅可比矩阵
```

**这个性质意味着什么？**

- ✅ **有解析解**：不用采样，直接算！
- ✅ O(1) 复杂度：每个高斯一次投影就搞定
- ✅ **保持连续性**：高斯本身就是连续函数

### 3DGS vs NeRF：表示的根本区别

| 维度 | NeRF | 3DGS |
|------|------|------|
| **表示形式** | MLP权重（隐式，黑盒）| 高斯参数（显式，可读）|
| **查询方式** | 输入坐标→前向传播（慢）| 直接读取参数（快）|
| **渲染管线** | Ray Marching采样→积分 | 投影→排序→Alpha Blending |
| **每像素计算量** | ~100次MLP查询 | O(覆盖该像素的高斯数) |

### 速度为什么提升这么多？

回到我们的 1920×1080 场景：

```
NeRF: 
- 2M像素 × 100采样 = 2亿次MLP查询

3DGS（典型场景）:
- 约 1M 个高斯（根据场景复杂度调整）
- 每个高斯投影一次 → O(1)计算
- **总操作数**：~1M 次投影

对比：**100倍差距**（2亿 vs 1百万）
```

而且这还没算上：
- MLP 查询的内存开销 vs 直接读参数
- Ray Marching 的分支预测失败 vs 批量投影

### 实际性能对比

| 方法 | 训练时间 | 推理速度 | PSNR（质量） | 显存占用 |
|------|----------|----------|--------------|----------|
| NeRF（原始） | 1-2天 | **30秒/帧** | ~32 dB | <1GB |
| Instant-NGP | 30分钟 | 10-50ms/帧 | ~32 dB | ~2GB |
| **3DGS** | **30分钟** | **<10ms（60+fps）** | **~34 dB** | **~150MB** |

**直观理解**：3DGS 比原始 NeRF 快**3000倍**，质量还更好。

## 六、回顾：完整的问题演化链

现在让我们把整个故事串起来——**每一代的"遗留问题"都是下一代的起点**：

```
┌─────────────┐
│ 体素        │ ✓ 稠密表示，连续渲染
│ (Voxel)     | ✗ 内存立方增长（512³ → 4GB）
└──────┬──────┘
       │ ↓ "大部分空间是空的"
┌─────────────┐
│ 点云        │ ✓ 稀疏，只存表面
│ (Point Cloud)│ ✗ 点是零维的，投影有空隙
└──────┬──────┘
       │ ↓ "需要体积来贡献颜色"
┌─────────────┐  
│ 尝试给点    │ → Point Sprites（几何错误）
│ 加体积      | → 2D Splatting（无3D约束）
└──────┬──────┘       都失败了
       ↓ "需要既有体积又有正确几何"
┌─────────────┐
│ NeRF        │ ✓ 连续函数，质量极高
│             | ✗ MLP查询慢，ray marching离散
└──────┬──────┘
       │ ↓ "结构性瓶颈：隐式表示→必须采样"
┌─────────────┐
│ 3DGS        │ ✓ 显式高斯+投影闭式解
│ (突破!)     | ✓ 实时渲染，质量不输NeRF
└─────────────┘
```

### 核心洞察

**3DGS 的突破不是"优化 NeRF"**——它是识别了问题的本质矛盾：

> **隐式表示（MLP）+ 连续积分 = 结构性慢**

然后做了一个看似简单但关键的跳跃：**换到显式表示，利用高斯的投影封闭性**。

这个想法需要两个认知前提：
1. 接受点云作为起点（稀疏性已验证有效）
2. **意识到"3D 高斯 → 2D 高斯"的数学性质可用**

第二点才是关键——不是所有人都能从"需要体积"跳到"用高斯表示体积"。这需要知道：
- 高斯的投影封闭性（数学知识）
- Alpha Blending 可以混合多个半透明高斯（图形学直觉）

## 七、本章核心记忆点

✅ **视图合成问题** = 从有限照片重建可渲染的 3D 场景  
✅ **体素的问题**：内存立方增长，无法扩展到高分辨率  
✅ **点云的问题**：零维物体无法贡献颜色（采样间隙）  
✅ **NeRF 的质量来源**：MLP 作为通用函数逼近器 → 连续表示  
✅ **NeRF 的结构性瓶颈**：隐式表示 + ray marching = 每帧必须重新计算所有采样点  
✅ **3DGS 的关键跳跃**：**显式高斯 + 投影闭式解** → O(1) 渲染每个 primitive  
✅ **性能提升**：比 NeRF 快 3000 倍（30秒→0.01秒），质量更高

想想哪些场景可以接受 30 秒/帧的推理速度...

<summary>提示</summary>
离线渲染：电影特效、产品可视化（一次性生成）
</details>

### Q2: "3DGS 的 150MB 内存是怎么来的？"

假设 1M 个高斯，每个存储哪些参数？试着算一下...

<details>
<summary>提示</summary>
- 位置 μ: 3 个 float (x, y, z)
- 颜色 c: 3 或 5 个 SH 系数
- 协方差 Σ: 用 3 个缩放 +4 个四元数（避免对称矩阵的冗余）
- 不透明度 α: 1 个值  
→ 大约 52 字节/高斯，1M ≈ 52MB。实际 150MB 是因为...？
</details>

### Q3: "3DGS 牺牲了什么来换取速度？"

NeRF 可以表达视角相关的颜色（镜面反射）。3DGS 能吗？为什么？

<details>
<summary>思考方向</summary>
- NeRF：输入 (x, d) → c(x,d)，显式建模视角依赖
- 3DGS：每个高斯的颜色是固定的...但有 View-Dependent Appearance！下一章会讲到如何加入视角相关项。
</details>
