# 第1章：背景与问题定义 - 为什么3D重建需要新思路？

**学习框架**：`problem → starting point → invention → verification → example`

**核心目标**：明确3DGS要解决的真问题，理解NeRF的贡献与局限

---

## 一、问题：3D重建到底在干嘛？

### 1.1 一句话定义

从多张照片里重建出3D场景，然后能像玩游戏一样从任意角度渲染新照片。

**核心公式**（视图合成）：
```
已知：M张图片 + 每张的相机参数
目标：找到一个场景表示S
要求：用S + 新相机参数 = 渲染出接近真实的新图片
```

**魔鬼细节**：这个"Render"函数怎么写？不同的S导致完全不同的实现方式。

---

### 1.2 性能对比：为什么3DGS这么火？

**关键数据**（推理速度对比）：

| 方法 | 训练时间 | 推理时间/帧 | PSNR | 内存 |
|------|----------|-------------|------|------|
| NeRF（原始） | 1-2天 | **30秒** | 32-33 dB | <1GB |
| NeRF++ | 6-12小时 | 10秒 | 33-34 dB | <1GB |
| Instant-NGP | 30分钟 | 10-50ms | 32-33 dB | ~2GB |
| **3DGS** | **30分钟** | **<10ms** | **33-35 dB** | **~150MB** |

**直观对比**（对数尺度）：

```
速度轴 (秒/帧)
   ^
 30 |                    NeRF
    |
  1 |          NeRF++
    |
0.05|     Instant-NGP  3DGS
    |
0.01|___________________________
     体素  点云  NeRF  3DGS
```

**核心洞察**：3DGS在推理速度上有**100-3000倍**提升，同时保持相似质量（PSNR差距仅1-2dB）。

**问题来了**：为什么3DGS能跳脱"ray marching + MLP"这个慢速范式？

## 二、从射线出发：渲染的基础

### 2.1 射线的直觉理解

**一句话**：从相机光心出发，穿过每个像素的"视线"。

```
相机光心 O
   ↑
   | 射线 r(t) = O + t·d
   |
   • 穿过像素 (u,v)
   |
   V 延伸到无穷远，寻找"第一个挡住的东西"
```

**为什么需要射线？**
- 渲染本质：从相机"看"出去，每个像素一条视线
- 确定"看到什么"：沿射线找到最近的表面
- 体积渲染：沿射线累加颜色和密度（如果有半透明物体）

---

#### 🔬 **第一性原理：为什么需要 ray marching？**

**公理**：
1. 渲染是 3D→2D 的投影
2. 像素颜色由"可见表面"决定
3. 场景是连续的（可能有雾、玻璃等半透明）

**矛盾**：
- 我们需要知道射线与场景的"交点"
- 但场景是连续函数（如 NeRF 的 MLP），没有显式表面
- 因此必须**采样**多个点来近似积分

**解决路径**：
```
沿射线等距采样 t₁, t₂, ..., tₙ
对每个点 x(t_i) 查询场景属性 (color, density)
用 volume rendering 公式累加贡献
```

**Ray marching 算法**（通用模板）：
```python
C = 0  # 累计颜色
T = 1  # 透射率
for i in range(N_s):
    t = t_near + i * Δt
    x = O + t * d
    color, density = query_scene(x)  # 查体素或跑MLP
    alpha = 1 - exp(-density * Δt)
    C += T * alpha * color  # 前向混合
    T *= (1 - alpha)        # 透射衰减
    if T < 1e-4: break      # 完全遮挡就停
return C
```

**复杂度分析**（为什么NeRF慢）：
```
每帧像素数：1920×1080 ≈ 207万
每像素采样：N_s ≈ 100-200
总采样点数：207万 × 100 ≈ 2亿

NeRF的MLP查询：每采样点 ≈ 10,000 FLOPs
总计算量：2亿 × 1万 ≈ 2×10¹² FLOPs = 2 TeraFLOPs!

理论下限（RTX 4090，80 TFLOPS）：
  2 / 80 = 0.025秒 = 25ms

实际NeRF：30秒/帧
差距：约1200倍（内存访问、非峰值算力）

结论：MLP查询 + ray marching 是结构性瓶颈
```

**三个致命问题**：
1. **MLP查询慢** - 无法通过硬件线性加速，每帧必须重新算
2. **ray marching本质离散** - 采样越多越准但越慢
3. **训练时间长** - 每帧都要跑完整管线，动辄几十小时

### 2.2 从像素到世界：相机模型的核心

**核心问题**：怎么从像素坐标(u,v)得到世界空间的射线？

**变换链**（必须按这个顺序）：
```
像素 (u,v)
  → K⁻¹ → 归一化相机坐标 (x',y',1)
  → R   → 世界坐标方向 d_world
  → 射线 r(t) = O + t·d_world
```

**为什么是这个顺序？**
- K⁻¹ 作用在像素空间（齐次坐标）
- R 作用在3D向量（相机系的基向量）
- 两者在不同空间，不可交换

**验证**：掩住答案，从"已知(u,v)，求世界方向"出发：
- 需要从像素尺度→角度尺度 → 除以焦距 → K⁻¹
- 需要从相机系→世界系 → 旋转矩阵 R
- 能否用R·K？❌ 旋转后像素无意义
- 能否用K⁻¹·R？❌ 旋转矩阵不能作用在2D像素
- **结论**：`d = R·K⁻¹·[u,v,1]ᵀ` 是唯一正确链

---

#### 🔬 **内参K：像素的"翻译官"**

**标准形式**（针孔模型）：
```
K = [f_x,  s, c_x]
    [ 0, f_y, c_y]
    [ 0,   0,   1]
```

**参数意义**：

| 参数 | 是什么 | 典型值 | 作用 |
|------|--------|--------|------|
| f_x, f_y | 焦距（像素单位） | 1000-2000 | 除以焦距 → 角度尺度 |
| c_x, c_y | 主点（图像中心） | W/2, H/2 | 减去主点 → 以光心为原点 |
| s | 切向畸变 | 0 | 非方形像素修正 |

**K⁻¹的关键计算**（s=0时）：
```
K⁻¹ = [1/f_x,     0,     -c_x/f_x]
      [   0,   1/f_y,    -c_y/f_y]
      [   0,     0,         1    ]
```

**验证特例**：
- 中心像素 `(c_x, c_y)` → `[0,0,1]ᵀ` → 沿光轴 ✅
- 左上角 `(0,0)` → `[-c_x/f_x, -c_y/f_y, 1]ᵀ` → 向左上倾斜 ✅

**K为什么是上三角？**
从投影公式 `u = f_x·(x/z) + c_x` 反推：
- f_x, f_y 在(1,1),(2,2)：缩放x,y
- s在(1,2)：x,y耦合（非方形像素）
- c_x, c_y在(1,3),(2,3)：平移
- 下三角为0：Z方向无耦合（针孔假设）

**畸变呢？**
- 径向畸变：在投影后应用，不在K内
- 完整流程：像素 → 去畸变 → K⁻¹ → 相机方向

---

#### 🔬 **外参R：相机的"朝向"**

**定义**：R是从相机坐标系到世界坐标系的旋转。

**旋转矩阵性质**（SO(3)）：
- 正交性：`Rᵀ·R = I`（列向量正交单位）
- 行列式：`det(R) = +1`（保持手性）
- 逆：`R⁻¹ = Rᵀ`

**物理意义**：
```
R的行向量 = 相机轴在世界系的表示:
  R[0,:] = X_camera_in_world  (相机X轴)
  R[1,:] = Y_camera_in_world  (相机Y轴)
  R[2,:] = Z_camera_in_world  (相机Z轴)
```

**常见朝向**：
```
1. 相机看向+Z (NeRF标准):
   X_camera=(1,0,0) 右
   Y_camera=(0,1,0) 下
   Z_camera=(0,0,1) 前
   R = I

2. 绕Y轴旋转90° (看向+X):
   R_y(90°) = [0, 0, 1]
              [0, 1, 0]
              [-1,0, 0]
   验证: R·[0,0,1]ᵀ = [1,0,0]ᵀ ✅
```

**为什么用矩阵而不是欧拉角？**

| 表示 | 维度 | 奇异性 | 优化友好度 | 投影复杂度 |
|------|------|--------|------------|------------|
| 旋转矩阵R | 9维 | 无 | SVD修复 | R·v 直接乘 |
| 欧拉角 | 3维 | 有（万向锁） | 简单 | 需转矩阵 |
| 四元数 | 4维 | 无 | 需归一化 | q→R 多一步 |

**NeRF/3DGS用R的原因**：
- 无奇异性（欧拉角在俯仰90°失效）
- 梯度稳定（9维优化，SVD投影约束）
- 投影简单（直接矩阵乘法）
- 现代GPU算矩阵快

**实际做法**：
```python
# 优化9维矩阵M，投影时强制正交
R = svd(M)[0]  # 或 Gram-Schmidt
```

---

#### 🔬 **方向向量需要归一化吗？**

**问题**：`d = R·K⁻¹·[u,v,1]ᵀ` 的长度是1吗？

**分析**：
```
K⁻¹·[u,v,1]ᵀ = [x', y', 1]ᵀ
长度 = sqrt(x'² + y'² + 1) ≥ 1

R是正交矩阵: ||R·v|| = ||v||
所以 ||d|| ≠ 1（除非x'=y'=0）
```

**物理意义**：
- d的方向正确，但长度≠1
- 参数t的尺度：`r(t) = O + t·d`
- t=1 → 移动 `||d||` 距离（不是1米）

**体积渲染中t通常用物理距离 → 需要归一化**：
```python
d = R @ (K⁻¹ @ pixels)
d = d / np.linalg.norm(d, axis=0)  # 归一化
```

**为什么公式通常不写归一化？**
- 简洁
- 实际实现时归一化
- 数学上方向正确即可，长度可缩放

### 2.3 完整射线生成流程（实战代码）

**伪代码**：

```python
def generate_rays(camera, H, W):
    """
    输入:
      camera: {K: 3x3, R: 3x3, O: 3x1}
    输出:
      rays: {origin: 3x1, direction: 3x(H*W)}
    """
    # 生成像素网格
    u = np.arange(W)
    v = np.arange(H)
    uu, vv = np.meshgrid(u, v)  # H×W

    # 齐次像素坐标 (3×N)
    pixels = np.stack([uu.ravel(), vv.ravel(), np.ones(H*W)], axis=0)

    # K⁻¹变换 → 相机系方向
    dirs_camera = K⁻¹ @ pixels  # 3×N

    # 归一化（Z=1平面）
    dirs_camera = dirs_camera / dirs_camera[2,:]

    # R旋转到世界坐标
    dirs_world = R @ dirs_camera  # 3×N

    # 射线原点（所有像素共用）
    origins = O.reshape(3,1)

    return {'origin': origins, 'direction': dirs_world}
```

**向量化**：所有像素同时变换（矩阵乘法批量），GPU友好。

### 2.4 坐标变换链的完整数学

**正向（3D→2D）**：
```
世界坐标 X_w
    ↓  X_c = R·(X_w - O)  (刚体变换)
相机坐标 X_c
    ↓  [x/z, y/z, 1]ᵀ  (透视投影)
归一化设备坐标 (NDC)
    ↓  [u, v, 1]ᵀ = K·[x/z, y/z, 1]ᵀ
像素坐标 (u,v)
```

**反向（射线生成）**：
```
像素 (u,v)
  → K⁻¹ → [x', y', 1]ᵀ  (相机系归一化方向)
  → R   → d_world = R·[x',y',1]ᵀ  (世界方向)
  → r(t) = O + t·d_world  (射线)
```

**齐次坐标的作用**：
- 统一线性变换（平移、旋转、投影）
- K是3×3，作用在齐次像素[u,v,1]ᵀ
- 结果仍是齐次，需除以第3分量得到2D

**为什么R不包含平移？**
- 射线方向只依赖旋转（平移不影响方向）
- 平移单独作为射线原点
- 分解：方向 = R·(相机方向)，原点 = O

**核心要点总结**（记忆卡片）：

| 概念 | 数学 | 物理意义 | 为什么重要？ |
|------|------|----------|--------------|
| K内参 | 3×3上三角 | 焦距(f_x,f_y)、主点(c_x,c_y) | 像素→相机坐标 |
| K⁻¹ | 上三角逆 | 撤销内参，得归一化方向 | 射线方向第一步 |
| R外参 | SO(3)正交 | 相机在世界系的朝向 | 相机→世界坐标 |
| R性质 | Rᵀ=R⁻¹ | 旋转保持长度 | 数值稳定 |
| 变换链 | d=R·K⁻¹·[u,v,1]ᵀ | 像素→世界方向 | 射线生成基础 |

## 三、体素：第一反应与内存墙

### 3.1 体素的工作原理

**思路**：把3D空间切成小立方体，每个存颜色和密度。

```
3D空间  →  划分为 V×V×V 体素网格
              ↓
每个体素存储 occupancy/color
              ↓
Ray marching: 沿ray采样t
              ↓
Trilinear插值读取体素值
              ↓
累加颜色得到像素
```

**数学公式**：
```
设体素网格函数: V(x,y,z) ∈ [0,1] (occupancy) 或 RGB
ray参数化: p(t) = o + t·d, t∈[t_near, t_far]
渲染: C = ∫ T(t)·α(p(t))·c(p(t)) dt
     离散化: C ≈ Σ T_i·α_i·c_i·Δt_i
```

### 3.2 内存墙：立方增长的数学必然

**内存计算表**：

```
+--------+-----------+----------------+-----------+
| 分辨率 | 体素数    | 内存 (float32) | 可行性    |
+--------+-----------+----------------+-----------+
| 256³   | 16.7M     | 64 MB          | ✅ 可行    |
| 512³   | 134M      | 512 MB         | ✅ 可接受  |
| 1024³  | 1.07B     | 4 GB           | ⚠️ 边缘   |
| 2048³  | 8.59B     | 32 GB          | ❌ 不可行 |
+--------+-----------+----------------+-----------+
```

**增长规律**：
```
内存 ∝ V³
V提高2倍  →  内存增加 8倍
V提高4倍  →  内存增加 64倍
```

**为什么是立方？**
- 1D: 长度L → 内存O(L)
- 2D: 面积L² → 内存O(L²)
- 3D: 体积L³ → 内存O(L³)
这是维度增加的数学必然，无法绕过。

**质量天花板**：离散网格 + 插值 → 边界必然有锯齿。

```
理想光滑表面:  ~~~~~~~~~~~~~~~~~~~~~~
                (连续曲线)

体素表示表面:  `````````````````````
                (离散台阶)
```

**结论**：体素在"快速查询"约束下是必然选择，但内存墙和质量天花板使其**不可扩展**。

## 四、点云：稀疏性的觉醒

### 4.1 核心洞察：99.9%的空间是空的！

这是3D表示方法的**范式转变**。

```
3D空间占用率估算 (真实场景):
+----------------------+--------+--------+
| 区域类型             | 占比   | 说明   |
+----------------------+--------+--------+
| 空气/空区域          | 99.9%  | 无物体 |
| 物体表面             | 0.1%   | 有物体 |
+----------------------+--------+--------+

推论: 为什么要存储空体素? 只存表面点!
```

**内存对比**：
- 体素: V³ (如512³ = 134M)
- 点云: N (表面点数, 通常10⁵-10⁶)
- **内存减少: 100-1000倍** ✅

**稀疏性是硬约束**，必须保留。

---

### 4.2 点云渲染的"三次尝试"

点云知道"哪里有表面"，但不知道"表面有多大"。三次尝试都在解决"如何给点加体积"。

---

#### 尝试1: 点精灵 (Point Sprites)

```
方法: 每个点画一个面向相机的方块 (Billboarding)

问题:
  • 点之间有空隙 → 不连续
  • 方块角度固定 → 不是真3D
  • 放大后明显看到方块边界

结果: ❌ 质量差
```

**诊断**：方块有面积，但**面积固定且不随视角变化** → 不是真3D体积。

---

#### 尝试2: Splatting (2D高斯)

```
方法: 每个点画一个高斯模糊圆 G(x) = exp(-|x-μ|²/2σ²)

问题:
  • 这个"高斯"是2D屏幕空间的 → 不是真3D
  • 只是"贴纸"，没有3D几何约束
  • 不同视角下，同一个3D点应该投影到不同大小的2D高斯 → 做不到

关键问题: 2D高斯的尺度σ如何确定?
  • 如果固定σ: 近大远小 → 透视错误 ✗
  • 如果根据深度调整: 需要知道3D"体积"
    → 点云没有3D体积信息! ✗
```

**诊断**：2D高斯缺少**3D几何约束**，无法正确表达透视变化。

---

#### 尝试3: 表面重建 (Poisson → Mesh)

```
方法: 点云 → Poisson重建 → Mesh → 三角形渲染

结果:
  • 质量提升 ✅
  • 但过程复杂 ❌ (Poisson重建慢)
  • 丢失细节 (Mesh简化) ❌
  • 无法端到端优化 ❌
```

**诊断**：Mesh 质量好，但**失去了稀疏性和可微性**，是"退步"。

---

### 4.3 点云的"最后一公里"问题

**问题诊断**：

```
点云: 位置 + 颜色
    → 渲染需要: 每个像素有3D元素"负责" → 需要面积
    → 质量需要: 连续投影 → 需要连续概率分布
    → 速度需要: 直接投影 → 需要解析公式

缺口: 点云有位置，但没有"影响范围"（体积）
```

**突破口的思考**（关键洞察）：

```
如果在3D空间给每个点定义一个"影响范围"（椭球），
让它自然投影到2D，会怎样?

3D点 + 3D椭球体积  →  投影  →  2D高斯
        ↑
    这就是3DGS的核心思想
```

**需要细化吗？** 这个"3D体积自然投影到2D"是3DGS的eureka moment。想深入讨论为什么是椭球吗？

---

#### 🔬 **第一性原理：为什么需要"体积"？**

**公理**：
1. 渲染需要面积覆盖（每个像素必须有3D元素负责）
2. 质量需要连续（不能有锯齿或点状伪影）
3. 点云是稀疏的（内存O(N)）

**矛盾**：
- 点云是零维的（没有面积）❌
- 直接投影到2D导致点间空隙（不连续）❌
- 但点云稀疏性好（内存O(N)）✅

**解决路径**：
```
给每个点定义一个"影响范围"（3D体积）
这个体积投影到2D应该是连续的（如高斯）
体积形状应能表示各向异性（拉长的椭球）
```

**候选形状对比**：

```
形状      | 投影复杂度 | 边界连续? | 各向异性?
----------|------------|-----------|----------
立方体    | 复杂       | ❌ (六边形) | ✅
球体      | 简单       | ✅         | ❌ (只能各向同性)
椭球      | 中等       | ✅         | ✅
高斯      | 简单       | ✅ (解析)  | ✅ (协方差)
```

**为什么高斯是唯一解？** 下一章详细推导。

**点云的遗产**：
- ✅ 证明了**稀疏性**可行（内存减少100倍）
- ✅ 证明了**直接投影**可以快速（无需ray marching）
- ❌ 遗留问题：零维度 → 需要给点加体积

## 五、NeRF：连续表示的代价

### 5.1 NeRF的核心思想

**公式化表示**：
```
场景表示为神经网络:
  F_θ(x, y, z, direction) → (color, density)

渲染 (ray marching):
  C(r) = ∫_{t_near}^{t_far} T(t)·σ(t)·c(t) dt
  其中 T(t) = exp(-∫_{t_near}^t σ(s) ds)
```

**训练**：
- 每张图像: 采样ray，计算渲染颜色
- 损失: L2(渲染, GT)
- 反向传播: 更新θ

**优点**：连续表示，质量好，视角依赖效果（镜面反射）建模成功。

---

### 5.2 NeRF的"三宗罪"

#### 罪1: MLP查询慢（无法避免）

**复杂度分析**：
```
每ray采样数: N_s ≈ 100-200
每样本MLP FLOPs: ≈ 10⁴
每ray FLOPs: N_s × 10⁴ ≈ 10⁶

图像ray数: N_p ≈ 2M (1920×1080)
总FLOPs/帧: N_p × N_s × 10⁴ ≈ 2×10¹² = 2 TeraFLOPs!

理论下限 (RTX 4090, 80 TFLOPS): 25ms
实际 NeRF: 30秒/帧
差距: 1000倍（内存访问、非峰值算力）
```

**为什么MLP不能缓存?**
- 每个ray采样点不同 → 无法重用
- 视角变化 → 采样点变化 → 必须重新计算
- **结构性缺陷**：3D信息编码在权重中，无法直接查询

---

#### 罪2: ray marching本质离散

**采样-质量权衡**（无解）：
```
采样点N_s ↑ → 质量↑ 但 速度↓
N_s=64:  快但噪点多
N_s=128: 平衡
N_s=256: 质量好但慢 2 倍
```

**为什么必须采样？**
- 连续函数f(x)的积分无解析解（f是MLP）
- 必须数值积分 → 离散采样
- **这是数学必然，无法避免**

---

#### 罪3: 训练时间长

**原因**：
- 每样本都要MLP查询（慢）
- 每epoch所有图像都要跑完整管线
- 数据量: M=100-200张图, 每图2M ray → 每epoch 200M ray查询

**时间估算**：
```
每ray 1ms (含MLP) → 每epoch 200s
30k步 ≈ 1500 epochs → 300,000s ≈ 83小时
(实际有优化, 但仍需数小时-数天)
```

---

#### 🔬 **第一性原理：NeRF瓶颈的根源**

**公理**：
1. 场景是连续函数（MLP输入x,d输出color,density）
2. 渲染需要沿射线积分（体积渲染方程）
3. MLP是通用函数逼近器

**矛盾**：
- MLP每帧必须重新计算（无法缓存3D结构）
- 采样数N_s与质量正相关，但速度负相关
- 训练需要大量射线

**解决路径**：
```
问题：MLP查询慢 + ray marching慢
思路：能否避免ray marching？能否避免每样本MLP？

方向1：预计算3D结构 → 缓存（Instant-NGP: hash grid）
方向2：改变表示 → 可直接投影的primitive（3DGS: 高斯）
```

**验证**：
- 掩住答案，从"连续表示+实时渲染"推导：
  - 连续表示 → 需要积分 → 需要采样
  - MLP → 每样本计算 → 无法缓存
  - 能否预计算？ → Instant-NGP尝试，但仍有查询开销
  - **结论**：NeRF的瓶颈是"连续函数+实时查询"的**结构性矛盾**

---

**NeRF的遗产**：
- ✅ 证明了**神经连续表示**可行
- ✅ 视角依赖效果（镜面反射）建模成功
- ✅ 端到端可微训练范式
- ❌ 局限：MLP瓶颈、离散采样、无缓存 → 必须突破

## 六、演进总结：从体素到3DGS

### 6.1 设计约束的传递与突破

```
需求: 实时高质量渲染
  ↓
约束1: 快速投影 → 需要O(1)查询
  ↓
排除: NeRF(MLP慢)
  ↓
约束2: 连续质量 → 需要连续PDF
  ↓
排除: 体素(离散)、点云(δ函数)
  ↓
约束3: 稀疏内存 → N<<空间体积
  ↓
保留: 点云思想(稀疏)+高斯(连续+投影解析)

方案: 3D高斯椭球 + 直接投影 + Alpha Blending

## 七、思考题（独立重推）

### 1. Ray marching 的复杂度

**问题**：
- H=1920, W=1080, N_s=100，总采样数？
- 如果每采样 1μs，单帧需要多久？
- **重推**：从 volume rendering 积分公式，为什么需要离散采样？

**推导**：
```
总采样数 = 1920 × 1080 × 100 = 2.07亿
时间 = 2.07亿 × 1μs = 207秒 ≈ 3.5分钟

体积渲染积分: C(r) = ∫ T(t)·σ·c dt
σ和c是任意连续函数 → 无解析解 → 必须离散采样
```

**答案**：2亿采样，3.5分钟。离散采样的原因是 σ 和 c 是任意函数，无解析积分。

---

### 2. 体素内存计算

**问题**：
- V=2048 需要多少内存？可行吗？
- **重推**：内存 O(V³) 的数学来源？

**计算**：
```
V=2048 → 2048³ = 8.59B 体素 → 34.4 GB
消费级GPU最大24GB → ❌ 不可行

数学来源: 3D网格单元数 = V×V×V = V³
```

### 3. PSNR 与主观感受

**问题**：
- PSNR 差 1dB，人眼能察觉吗？
- **重推**：MSE 是像素级误差，为什么对数变换更符合人眼？

**分析**：
```
PSNR 每6dB → MSE减半
1dB ≈ MSE减少20%

人眼: 对低频（平滑区域）敏感，对高频（纹理）容忍
MSE→log变换: 大误差惩罚重，小误差惩罚轻 → 更符合感知
```
