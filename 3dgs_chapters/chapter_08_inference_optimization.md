# 第 8 章：为什么训练完的模型还是跑不动——推理优化到底在优化什么

**本章核心问题**：第 7 章已经解释了 3DGS 怎样被训练出来。现在问题变成：

> 为什么一套已经学会场景的 Gaussian，直接拿训练时那条前向链去跑，常常仍然慢得没法实时使用？推理优化到底在优化什么，是“换个 CUDA 技巧”，还是更深的计算结构问题？

如果前面几章回答的是：

- 第 3 章：为什么 primitive 选 Gaussian
- 第 4 章：Gaussian 怎样被渲染成图
- 第 5 章：优化目标为什么这样设计
- 第 6 章：第一批 Gaussian 从哪来
- 第 7 章：它们怎样被训练到收敛

那么这一章回答的就是：

```text
场景已经学会了
为什么还是跑不快
以及到底该从哪几层下手，把它变成实时推理系统
```

先把主线写在前面：

```text
训练前向和推理前向共享同一条渲染链
但训练还背着反向传播、结构编辑和各种诊断成本

推理优化的核心不是“换个写法”
而是重新审视渲染链里真正花时间的部分：
projection / sorting / tile mapping / blending / memory bandwidth

真正的目标不是把理论公式改掉
而是让同一条渲染链以更少的数据搬运、更少的重复计算、更少的无效覆盖运行
```

---

## 一、先分清：训练前向和推理前向到底哪里相同，哪里不同

### 1.1 相同的部分：渲染主链本身没有变

无论训练还是推理，核心前向链条都还是第 4 章那一条：

```text
3D Gaussian
-> 世界到相机变换
-> 局部投影成 2D footprint
-> 深度组织
-> tile 局部筛选
-> front-to-back alpha blending
-> 图像
```

也就是说：

> 训练和推理不是两套完全不同的渲染器，它们共享同一个成像骨架。

### 1.2 不同的部分：训练额外背着很多推理不需要的东西

训练时，你还要做：

- 反向传播
- 梯度缓存
- optimizer 更新
- densify / split / prune
- 各种监控、日志、可视化

而推理时真正需要的只剩下：

```text
给定相机位姿
尽快得到一张确定的图像
```

这说明一个非常关键的事实：

> 推理不是“把训练代码去掉 backward 就完事”，而是要重新针对延迟、带宽和吞吐做设计。

---

## 二、推理真正优化的是什么目标

训练时你最关心的是：

- 最终质量
- 收敛稳定性
- 可训练性

推理时你最关心的则是：

- 单帧延迟
- 吞吐量
- 内存带宽利用率
- 输出稳定性

### 2.1 单帧延迟为什么是硬约束

实时交互里，延迟不是一个“越小越好”的装饰指标，而是强物理约束。

- `16.7ms` 对应 60 FPS
- `11.1ms` 对应 90 FPS
- VR/AR 里常常希望更低

所以如果你训练完一个场景，渲染一帧还要 `150ms`，它不是“稍微慢一点”，而是：

```text
根本不在实时系统的工作区里
```

### 2.2 推理还要求确定性

训练时可以接受：

- 随机采样视角
- 梯度噪声
- 动态结构变化

推理时却要求：

- 同样相机位姿给出同样图像
- 相邻帧不要莫名闪烁
- 缓存策略不能引入明显不稳定

这意味着推理优化要非常注意：

> 哪些近似只是重排计算，哪些近似真的会改变输出，甚至引入抖动。

---

## 三、先把瓶颈拆开：一帧时间到底花在哪

为了不空谈优化，先把 3DGS 推理前向粗略拆成五部分：

```text
T_frame ≈ T_project + T_sort + T_tile + T_blend + T_memory
```

更准确地说：

- `T_project`：把 3D Gaussian 投到当前相机，得到 `mu_2d`、`Sigma_2d` 等
- `T_sort`：按深度组织前后顺序
- `T_tile`：建立 tile 到相关 Gaussian 的映射
- `T_blend`：逐 tile / 逐像素做 front-to-back blending
- `T_memory`：各种中间结果的读写和搬运成本

这五项不是完全独立的，但它们已经足够把问题结构看清楚。

### 3.1 一个粗略成本模型

如果高斯数是 `N`，像素数是 `P = H * W`，那么常见成本直觉大概是：

```text
projection:   O(N)
sorting:      O(N log N)
tile mapping: O(N * avg_tiles_per_gaussian)
blending:     O(P * avg_gaussians_per_pixel)
```

这条分解特别重要，因为它告诉你：

> 不同分辨率、不同高斯数量、不同场景局部复杂度下，瓶颈可能根本不是同一个地方。

---

## 四、第一个瓶颈：projection 为什么看起来简单，却不一定便宜

projection 这一步在数学上并不复杂。第 4 章已经写过：

```text
mu_cam = R * mu_world + t
Sigma_cam = R * Sigma_world * R^T
```

然后中心投影：

```text
u = fx * X / Z + cx
v = fy * Y / Z + cy
```

Jacobian：

```text
J = [[fx / Z, 0, -fx * X / Z^2],
     [0, fy / Z, -fy * Y / Z^2]]
```

2D 协方差：

```text
Sigma_2d ≈ J * Sigma_cam * J^T
```

### 4.1 数学简单，不代表吞吐上免费

问题在于这件事要对每个 Gaussian 都做一遍。

如果 `N` 是几十万、上百万，那么即便每个 Gaussian 只是：

- 几次矩阵乘法
- 几次除法
- 一次 `2 x 2` 协方差构造

总成本也会堆起来。

### 4.2 projection 的真正风险不是公式复杂，而是内存访问太多

真实实现里常见的问题不是“算不出来”，而是：

- 读一大堆 Gaussian 参数
- 生成一大堆中间结果
- 再把这些结果写回显存
- 后续 tile / blend 又要再读它们一遍

所以 projection 之后，你立刻要问的不是：

```text
这个公式能不能再推简一点
```

而是：

```text
这些中间结果会不会被反复读写，导致带宽成为真正瓶颈
```

---

## 五、第二个瓶颈：排序为什么常常又必要又昂贵

### 5.1 为什么排序不能轻易省

第 4 章已经解释过，front-to-back alpha blending 依赖前后顺序。

所以从渲染正确性上看，你通常至少需要一种近似深度顺序：

```text
按 Gaussian 中心深度 z 排序
```

这并不完美，但在 Gaussian 足够局部时通常够用。

### 5.2 排序的开销长什么样

如果每帧都对全部 `N` 个 Gaussian 排序，粗略上是：

```text
T_sort ~ O(N log N)
```

当 `N` 很大时，这一步会肉眼可见地咬掉预算。

### 5.3 为什么排序缓存会成为第一批很自然的优化

这里就出现一个特别有工程味的观察：

> 如果相机只移动了一点点，深度顺序真的会完全重排吗？

很多时候不会。尤其是连续视角、小幅平移、慢速旋转时，大部分 Gaussian 的前后关系相对稳定。

于是很自然就会得到一种缓存思路：

```text
如果相机变化不大
就复用上一帧的排序结果
```

### 5.4 这类优化到底在赌什么

它赌的不是“排序不重要”，而是：

- 小视角变化时，排序变化往往是局部而不是全局
- 复用排序的误差通常小于完全重排带来的时间成本

所以这类缓存优化的本质是：

> 用相邻帧之间的时间连续性，换掉一部分重复全量排序。

### 5.5 但排序缓存一定有失效条件

这类缓存并不是永远可用。常见失效条件会是：

- 平移太大
- 旋转太大
- 局部遮挡关系显著变化

所以一个更准确的工程表述是：

```text
排序缓存 = 在相机小幅运动条件下，近似复用前一帧的深度组织结果
```

它不是“永远正确”，而是一个有边界条件的加速策略。

---

## 六、第三个瓶颈：tile mapping 为什么和 sorting 一样值得缓存

### 6.1 tile-based 渲染已经把问题局部化了，但每帧建映射本身也要花钱

第 4 章已经说过，Gaussian 最大的工程红利之一就是它是局部的，所以我们可以：

- 先算 2D footprint 包围盒
- 再把 Gaussian 分配到覆盖到的 tiles

这一步已经把全局像素 x 全局 Gaussian 的问题，压成了局部 tile x 局部 Gaussian。

但别忘了：

> 建立这个 `tile -> relevant gaussians` 映射，本身也是一笔真实开销。

### 6.2 为什么 tile 映射同样存在时间连续性

如果相机移动很小，很多 Gaussian 在屏幕上的 footprint 虽然会轻微移动，但通常：

- 仍在相邻 tile 附近
- 大部分还留在原有 tile 集合里
- 只有边界附近的部分 Gaussian 会跨 tile 变化

所以 tile mapping 也天然带时间连续性。

### 6.3 这意味着什么

意味着 tile mapping 和 sorting 很像：

```text
都不是每帧一定要完全从零重建
都可以在相机小幅变化时尝试复用或局部更新
```

所以一个特别自然的工程设计就是：

> 让排序缓存和 tile 缓存共享相近的失效逻辑。

这会让整条前向在“连续视角”场景里快很多。

---

## 七、第四个瓶颈：blending 为什么常常是最终逃不掉的硬活

### 7.1 blending 的成本随什么增长

对于每个像素，blending 成本大致和：

```text
该像素真正参与混合的 Gaussian 数
```

有关。

于是总成本粗略可以看成：

```text
T_blend ~ O(P * avg_gaussians_per_pixel)
```

这件事很重要，因为它告诉你：

- 分辨率翻倍，像素数暴涨
- 局部场景很复杂时，每像素参与混合的项也会增加

### 7.2 blending 为什么不像排序那样容易缓存

排序和 tile mapping 主要是“组织数据”。

blending 则是：

> 真正把当前视角下、当前像素下的这些局部贡献算成最终颜色。

只要视角一变、像素采样点一变，很多 blending 结果都得重新算。

所以这部分往往是推理里真正的“硬计算”层。

### 7.3 能对 blending 做的优化，更多是减少无效计算

你常见的加速方向通常不是“彻底跳过 blending”，而是：

- 让每像素面对更少的相关 Gaussian
- 提前终止（当剩余透射率已经很小）
- 让局部内存布局更适合 GPU 连续访问
- 减少中间结果反复搬运

所以如果把推理优化的最后一公里压成一句话，就是：

> 真正难省的那部分往往不是数学，而是每像素局部混合本身；优化重点是尽量减少它前后的无效成本。

---

## 八、第五个瓶颈：很多时候你真正卡的是带宽，不是算力

这一步特别容易被误判。

很多人看到推理慢，第一反应是：

```text
是不是公式太多了
是不是算子太重了
```

但大量 3DGS 实际瓶颈常常更像：

> GPU 不是算不动，而是在等数据。

### 8.1 为什么会这样

假设每个 Gaussian 存：

- 中心
- 形状参数
- opacity
- SH / 颜色参数

当 `N` 很大时，每帧都要从显存里读出大量结构。

如果这些数据：

- 布局分散
- 重复读取
- kernel 之间反复写回 / 再读入

那么真正卡住你的，往往不是 FLOPs，而是：

```text
memory bandwidth
```

### 8.2 这也是为什么混合精度常常收益这么直接

如果把很多推理阶段的存储从 `float32` 压到 `float16`，最直接的变化是：

- 存储减半
- 显存读写压力减半
- 某些 GPU 核心吞吐更高

只要不把数值稳定性搞崩，这种收益往往非常直接。

### 8.3 但哪些部分最适合降精度，哪些不该乱动

推理里常见的经验判断通常是：

- 存储层尽量压缩：`mu`、颜色、部分中间结果可考虑 `FP16`
- 对数值更敏感的环节要更谨慎：例如 `Sigma_2d` 求逆、极小深度下的 Jacobian 计算

所以更稳妥的策略通常不是：

```text
全链路无脑 half
```

而是：

```text
对带宽敏感的大块数据做低精度存储
对数值脆弱的小块计算保持更安全精度
```

这就是所谓 mixed precision 在 3DGS 推理中的真实工程意义。

## 九、几个典型优化，应该怎样放回“瓶颈 -> 收益 -> 代价”的框架里理解

这一节把常见优化按工程逻辑摆平，而不是当成技巧清单背。

### 9.1 `torch.no_grad()` / 去掉训练附加链

- **瓶颈**：训练前向额外维护 autograd 图和训练逻辑
- **优化**：推理时关闭 autograd，只保留纯前向
- **收益**：立刻减少显存和不必要开销
- **代价**：没有代价，这是推理应做的基础动作

### 9.2 排序缓存

- **瓶颈**：每帧全量排序代价高
- **优化**：相机变化小时复用前一帧排序
- **收益**：连续视角里能明显省时间
- **失效条件**：大视角跳变、显著遮挡变化

### 9.3 tile 缓存 / 局部更新

- **瓶颈**：每帧重建 `tile -> gaussians` 映射
- **优化**：在相机小变化时复用或局部更新
- **收益**：减少重复组织数据的时间
- **失效条件**：footprint 跨 tile 大范围变化

### 9.4 mixed precision

- **瓶颈**：显存带宽和存储压力
- **优化**：部分存储 / 计算改用 `FP16`
- **收益**：带宽压力降低，某些硬件吞吐提升
- **失效条件**：数值敏感环节可能不稳

### 9.5 kernel fusion

- **瓶颈**：多个 kernel 之间重复读写中间结果
- **优化**：把相邻阶段尽量融合，减少 global memory 往返
- **收益**：降低中间搬运成本
- **代价**：实现复杂度显著上升，调试更难

### 9.6 SoA / 更友好的内存布局

- **瓶颈**：参数布局不利于连续读取
- **优化**：让同类型属性连续存储，方便 coalesced access
- **收益**：提高实际带宽利用率
- **代价**：代码结构更偏底层，抽象性变差

所以这章真正想给你的不是一张“优化技巧表”，而是一个判断框架：

> 每一个优化都应该能回答：我在打哪个瓶颈、它的收益来自哪、什么时候会失效、它付出的复杂度值不值。

## 十、为什么第 8 章的本质还是“工程上把问题压回规则结构”

如果你退远一点看，会发现这章和 [../linear_algebra_engineering_perspective.md](../linear_algebra_engineering_perspective.md) 其实在讲同一种工程判断。

推理优化做的事，不是发明新数学，而是不断把问题往更规则、更可控、更可复用的结构里压：

- 高斯让局部表示变得规则
- Jacobian 让局部投影传播变得规则
- tile 化让局部计算范围变得规则
- 缓存让相邻帧重复性被利用起来
- mixed precision 让带宽消耗变得更可控

也就是说，这章真正讲的不是“性能黑魔法”，而是：

> 在不改掉核心渲染公式的前提下，怎样把这条公式链改写得更适合真实硬件。

## 十一、两个具体工程场景：为什么同一套优化在不同地方效果会完全不同

### 11.1 VR 头显：排序缓存和 tile 缓存特别值钱

在 VR 里，相机位姿通常：

- 高频更新
- 相邻帧变化小
- 需要极低延迟

这时时间连续性特别强，所以：

- 排序缓存收益往往很大
- tile 缓存也更容易命中

这类场景最适合吃“连续视角复用”这波红利。

### 11.2 4K 大分辨率离线预览：blending 会重新成为核心瓶颈

如果分辨率特别高，像素数 `P` 会猛涨。

这时就算排序和 tile 已经优化得不错，真正压帧时间的也可能重新变成：

```text
每像素 blending 的绝对工作量
```

所以不要指望一种优化在所有场景里都永远最重要。

不同场景会把不同项重新推到最前面。

## 十二、一个最小可运行实验：把推理开销结构画出来

下面这段代码不跑真实 CUDA，只做一件特别有价值的事：

> 用 toy 成本模型看，当 Gaussian 数量或分辨率变化时，projection / sorting / tile / blending 哪一项会先变成主瓶颈。

```python
import numpy as np
import matplotlib.pyplot as plt


N_values = np.linspace(1e5, 3e6, 120)
P = 1280 * 720

# toy cost model，只用于理解趋势，不代表真实绝对时间
T_project = 1.0e-6 * N_values
T_sort = 2.2e-7 * N_values * np.log2(N_values)
T_tile = 5.0e-7 * N_values
T_blend = np.full_like(N_values, 0.9e-8 * P * 35)  # 假设平均每像素 35 个 gaussian

plt.figure(figsize=(8.5, 5.2))
plt.plot(N_values / 1e6, T_project, label='projection')
plt.plot(N_values / 1e6, T_sort, label='sorting')
plt.plot(N_values / 1e6, T_tile, label='tile mapping')
plt.plot(N_values / 1e6, T_blend, label='blending')
plt.xlabel('number of gaussians (millions)')
plt.ylabel('relative time (toy units)')
plt.title('How bottlenecks shift as Gaussian count grows')
plt.legend()
plt.tight_layout()
plt.show()
```

你应该观察到：

- `blending` 更像和分辨率、每像素局部复杂度强相关
- `projection`、`tile` 更接近线性随 `N` 增长
- `sorting` 会因为 `N log N` 结构，在大规模时越来越显眼

你还可以把 `P` 改成 `1920 * 1080` 或 `3840 * 2160`，看看谁会重新成为主瓶颈。

这段实验最重要的作用不是给你真实时间，而是帮你建立一个判断习惯：

```text
先拆成本结构
再判断该优化哪里
而不是见招拆招地堆技巧
```

## 十三、把整章压成一个最短心智模型

如果你只想记一条链，就记这个：

```text
训练前向和推理前向共享同一条 Gaussian 渲染链

    ↓

但推理不再关心 backward、densify、optimizer
而是开始关心：延迟、带宽、吞吐和稳定输出

    ↓

真正的瓶颈通常分布在：
projection / sorting / tile mapping / blending / memory bandwidth

    ↓

优化不是乱上技巧
而是针对具体瓶颈做具体动作：
小视角变化 -> 用缓存
带宽吃紧 -> 用 mixed precision / 更好布局
中间搬运太多 -> kernel fusion

    ↓

推理优化的本质
是在不改掉核心渲染公式的前提下
把这条公式链重写得更适合真实硬件
```

这就是第 8 章真正要你建立起来的推理优化视角。

### Q1: "Tile-based rendering 为什么比朴素遍历快？"

假设图像分辨率 $H \times W = 1024^2$，高斯数量 $N = 500,000$，估算两种方法的复杂度差异。

<details>
<summary>提示</summary>
- 朴素：每个像素遍历所有 N 个高斯 → O(HW·N)
- Tile-based：每个 tile 只处理覆盖它的高斯 → 每 pixel 只看 k 个 (k << N)
- 500K 高斯 × 1M 像素 = 5×10¹¹次计算 vs 可能只有 5×10⁶~10⁷
</details>

<details>
<summary>答案</summary>
**朴素遍历复杂度**:
$$O(H \cdot W \cdot N) = O(1024^2 \cdot 500,000) \approx O(5 \times 10^{11})$$
→ **5000 亿次计算**,GPU 也要跑很久

**Tile-based culling**:
- 屏幕分成 $16 \times 16$ tiles → ~4000 个 tile
- 每个高斯只覆盖少数几个 tile（根据 $\Sigma_2d$ 的 k-sigma 包围盒）
- 每个 pixel 只看自己所在 tile 的相关高斯

假设平均每个 tile 有 $k \approx 50$ 个相关高斯：
$$O(H \cdot W \cdot k) = O(1024^2 \cdot 50) \approx O(5 \times 10^7)$$
→ **5000 万次计算**,快了**100 倍**!

**核心**: Gaussian 的局部性 → 不用看全局，只看局部相关。
</details>

### Q2: "SH (Spherical Harmonics) 阶数怎么选？"

为什么默认用 order=1 (5 coefficients)?order=3(16 coeffs)会怎样？

<details>
<summary>提示</summary>
- SH 阶数越高 → 能表达更复杂的视角相关颜色（镜面反射）
- 但参数越多 → 内存和训练难度增加
- order=1: 5 coefficients, order=3: 16 coefficients
</details>

<details>
<summary>答案</summary>
**SH (球谐函数)**: 用于表达**视角相关的颜色**（view-dependent appearance）。

| Order | Coefficients | 能表达的效应 | 内存/高斯 |
|-------|-------------|-------------|----------|
| **0** | 1 | 无视角依赖（颜色固定） | ~52 MB/M |
| **1** | 5 | 基础漫反射 + 简单高光 | ~68 MB/M |
| **3** | 16 | 复杂镜面反射、各向异性 | ~130 MB/M |

**默认 order=1 的原因**:
- 大多数场景的视角依赖可以用 5 个系数近似
- 内存开销适中（~68 bytes/高斯 vs 48 bytes for fixed color）
- 训练收敛快，不易过拟合

**什么时候用 order=3?**
- **强镜面反射**: 金属、 glossy surfaces
- **各向异性材质**: 头发、丝绸等

**权衡公式**:
$$\text{总内存} \approx N \cdot (48 + 12 \cdot (\text{order}+1)^2) \text{ bytes}$$

3DGS 默认 order=1 是工程上的合理折衷。
</details>

### Q3: "4DGS 和静态 3DGS 的核心区别是什么？"

如果要在动态场景中使用，需要修改哪些部分？

<details>
<summary>提示</summary>
- 3DGS: $\boldsymbol{\mu}(t), \Sigma(t)$ 固定不变
- 4DGS: 引入时间参数 $t$ → $\boldsymbol{\mu}(\mathbf{x}, t), \dots$
- 需要额外模型预测每个时刻的高斯参数
</details>

<details>
<summary>答案</summary>
**静态 3DGS**:
$$\text{高斯集合} = \{ (\boldsymbol{\mu}_i, \Sigma_i, c_i, \alpha_i) \}_{i=1}^N$$
→ **所有参数固定**,只能重建单帧场景

**4DGS (动态 3DGS)**:
$$\text{高斯集合}(t) = \{ (\boldsymbol{\mu}_i(t), \Sigma_i(t), c_i(t), \alpha_i(t)) \}_{i=1}^N$$
→ **参数随时间变化**,需要预测函数

**核心修改**:
1. **时空参数化**: 用 MLP/变形网络预测 $\mathbf{x}(t) = f(\mathbf{x}_0, t)$
2. **额外监督**: 需要多时刻的视频帧作为训练数据
3. **物理约束**: 添加运动平滑性正则化，避免抖动

**内存代价**:
- 静态：~50 bytes/高斯
- 4DGS: ~100-200 bytes/高斯 (加上时间参数)

**应用场景**: 
- 3DGS → 静态场景（建筑、物体扫描）
- 4DGS → 动态场景（人走路、动物运动）
</details>
